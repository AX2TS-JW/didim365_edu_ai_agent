# 부동산 실거래가 AI 에이전트 — 1~2주차 학습 회고록

> AI Agent가 무엇이고 어떻게 만들고 어떻게 평가하는지, 고민과 실수 포함하여 정리한 회고록입니다.

---

## 1. 기획 — 왜 이 프로젝트인가?

### 궁극적인 목표

> **"지금 집을 사는 게 맞을까, 전세로 사는 게 맞을까?"**

이 질문에 답하려면 실거래가 수치뿐 아니라 전세가율 추이, 전문가 시장 전망, 정부 정책 동향까지 **종합 분석**이 필요합니다. 하지만 현실은 데이터가 API, PDF, 뉴스 등에 흩어져 있어 종합이 어렵습니다.

최종적으로 만들고 싶은 것:
```
"서울에서 매매가 나을까 전세가 나을까?"

→ [실거래가 에이전트]     매매가/전세가 조회, 전세가율 계산
→ [PDF 분석 에이전트]     KB부동산 리포트 + 국토연구원 보고서 요약
→ [뉴스 분석 에이전트]    최근 부동산 뉴스 트렌드 파악
→ [종합 리포트 에이전트]  위 3가지를 종합한 의사결정 보조 리포트
```

### 단계별 확장 계획

| 주차 | 만드는 것 | 최종 목표와의 관계 |
|------|-----------|-------------------|
| **1주차** | 실거래가 조회 에이전트 | 기초 데이터 수집 능력 확보 |
| **2주차** | 평가 파이프라인 | 에이전트가 정확한지 검증 체계 구축 |
| **3주차** | PDF 하이브리드 검색 + LangGraph | 전문가 리포트 분석 능력 추가 |
| **4주차** | 서브에이전트 협업 + 장기 기억 | 실거래가+PDF+뉴스 종합 리서치 에이전트 |

### 기획 시 고민

- **왜 부동산?** — 매매 vs 전세 의사결정이라는 개인적 필요 + 공공 API 무료 + 데이터 정형화로 검증 용이 + 매매/전세 구분이 도구 선택 학습에 적합
- **왜 상세 API?** — `해제여부(cdealType)` 필드로 취소 거래 필터링 필수. 상세가 일반의 상위호환
- **API 키 함정** — 반드시 **디코딩 키** 사용 (인코딩 키는 이중 인코딩으로 인증 실패)

## 2. 1주차: AI 에이전트 만들기

### Day 1 — ReAct 개념

ReAct(Reasoning and Acting) = LLM이 **추론→행동→관찰**을 반복하는 패턴.
Chain(고정 파이프라인)과 달리 Agent는 **상황 보고 알아서 판단**합니다.

```
[Reason] "매매 질문 → 매매 도구"  →  [Act] API 호출  →  [Observe] 결과 수신  →  [Reason] "답변 생성"
```

`create_agent()`가 이 루프를 자동 처리하며, `astream(stream_mode="updates")`이 각 단계를 이벤트로 내보냅니다.

### Day 2 — 프로젝트 구조

```
[Frontend React]  ──POST /api/v1/chat (SSE)──▶  [Backend FastAPI]  ──astream()──▶  [Agent LangChain]
```

**SSE를 쓰는 이유**: ReAct 루프의 각 단계(판단→도구실행→최종답변)를 실시간으로 UI에 전달하기 위해. 일반 REST는 최종 답변만 한 번에 반환 가능.

### Day 3 — create_agent() 5가지 재료

```python
agent = create_agent(
    model=ChatOpenAI(model="gpt-4.1"),              # 두뇌
    tools=[search_apartment_trades, ...],            # 손 (API 호출)
    system_prompt=system_prompt,                     # 업무 지침서
    response_format=ToolStrategy(ChatResponse),      # 답변 양식 강제
    checkpointer=InMemorySaver(),                    # 대화 기억
)
```

| 빠지면? | 결과 |
|---------|------|
| Model | 자연어 해석 불가 |
| Tools | 실제 데이터 조회 불가 → 할루시네이션 |
| System Prompt | 범위 제한 없이 광범위하게 동작 |
| Response Format | 구조화 파싱 불가 → 프론트엔드 깨짐 |
| Checkpointer | 멀티턴 불가 → 매번 맥락 상실 |

**ToolStrategy vs 프롬프트**: 프롬프트로 "JSON으로 답해줘"는 형식이 깨질 수 있지만, ToolStrategy는 LLM의 도구 호출 메커니즘으로 구조 **100% 보장**. `agent_service.py`가 `tool.get("name") == "ChatResponse"`로 파싱하므로 클래스명 변경 불가.

**@tool docstring = 도구 설명서**: LLM은 docstring을 읽고 어떤 도구를 쓸지 판단합니다.

**동적 날짜 주입**: `prompts.py`에서 `datetime.now()`로 현재 날짜를 시스템 프롬프트에 삽입 → 미래 월 조회 방지.

### Day 4 — Elasticsearch 캐시

ES를 API 응답 **캐시**로 활용: ES에 있으면 바로 반환, 없으면 API 호출 후 ES에 저장.

**지역코드 2단계 폴백**: ES 검색 → 하드코딩 딕셔너리. 단, "판교"→"분당구" 같은 동→시군구 매핑은 미지원 (2주차 진단에서 확인).

**해제 거래 필터링**: `cdealType` 존재 시 취소된 거래 → 제외해야 정확한 시세.

> **[개선] 캐시 TTL 추가**: 실거래가는 신고 지연으로 데이터가 계속 추가됨. 기존엔 1건만 있어도 캐시 히트 → 불완전한 옛 데이터 반환 문제. `_is_cache_expired()` 추가하여 현재 월 6시간 / 과거 월 7일 TTL 적용. 기존 `fetched_at` 필드 활용. 테스트 10건 통과.

### Day 5 — 통합 완성

**전체 흐름**: 프론트엔드 → `chat.py`(입력 필터링) → `agent_service.py`(에이전트 실행) → ReAct 루프 → `chat.py`(출력 필터링) → SSE → 프론트엔드

**3계층 보안**: A. 시스템 프롬프트(내부 공개 금지 지시) → B. 입력 필터링(정규식 인젝션 탐지) → C. 출력 필터링(유출 문구 검사)

**무한 루프 이중 방지**: 프롬프트("최대 3회") + 코드(`MAX_TOOL_CALLS = 3`). LLM은 지시를 100% 따르지 않으므로 코드로도 강제.

## 3. 2주차: 에이전트 평가하기

핵심 질문: **"에이전트가 왜 틀리는지를 어떻게 정량적으로 알 수 있는가?"**

### Day 1 — Opik 트레이싱

**관찰성**: 에이전트 내부 동작을 눈에 보이게 만드는 것.

```
트레이싱 적용 후:  질문 → Span 1: LLM 추론(2.1초) → Span 2: Tool 호출(3.5초) → Span 3: 답변 생성(1.8초)
```

`track_langgraph(agent, opik_tracer=tracer)` 한 줄로 모든 LLM/도구 호출이 자동 기록. 에이전트 생성 시점에 적용해야 하는 이유: 에이전트 내부 노드에 계측 코드를 심는 것이므로 HTTP 레이어에서는 불가.

### Day 2 — Trace 분석으로 진단

실패 유형별 진단: 지역코드 매핑 실패(Tool span), 면적 필터링 누락(포맷팅 단계), 미래 월 조회(LLM 추론 span).

진단 케이스 10종: 모호한 지역명, 미래 월, 복합 질의, 동→시군구 매핑, 프롬프트 인젝션 등.

**한계**: 사람이 하나씩 눈으로 봐야 함 → 대량 평가 불가 → LLM Judge 필요.

### Day 3 — LLM-as-a-Judge

LLM에게 **루브릭(채점표)**을 주고 1~5점 자동 채점. 에이전트(gpt-4.1)와 다른 모델(gpt-4o-mini)을 판사로 사용하여 독립성 확보.

`with_structured_output(JudgeResult)`로 점수+근거를 구조화 파싱 — 1주차 ToolStrategy와 같은 원리.

**한계**: 판사 LLM은 "정답"을 모름. 에이전트가 "20억"이라 했는데 실제 25억이어도 "잘 정리됨" → 높은 점수 가능. **숫자 정확성은 LLM만으로 검증 어려움** → Faithfulness 필요.

### Day 4 — DeepEval 다각도 평가

4가지 메트릭으로 **각기 다른 측면**을 평가:

| 메트릭 | 평가 내용 | 특징 |
|--------|-----------|------|
| AnswerRelevancy | 질문에 적절한 답변인가 | LLM 기반 |
| Faithfulness | 도구 결과에 충실한가 (할루시네이션 탐지) | LLM 기반 |
| GEval | 도구 선택이 적절한가 (커스텀 기준) | LLM 기반 |
| ToolUsage | 도구 선택, 횟수, 중복, 응답 시간 | **규칙 기반 (무료)** |

**핵심 사례**: "전세 시세" 질문에 매매 도구 호출 → Faithfulness만 높고 나머지 낮음. **메트릭 하나만 보면 오판, 4개 조합해야 원인과 영향 분리 가능.**

Faithfulness 평가를 위해 `call_agent_with_context()`가 도구 결과(retrieval_context)를 별도 수집.

### Day 5 — 종합 리포트

`generate_report.py`가 Day 2~4 결과를 합쳐 진단 리포트 + 개선 우선순위 자동 도출.

**2주차 평가 체계 발전 요약**:
| 전환 | 보완한 한계 |
|------|-------------|
| Trace → Judge | 수동→자동, 개별→대량 |
| Judge → DeepEval | 단일 관점→다각도, 문제 원인 분리 |

**남는 한계**:
1. **평가 데이터셋이 너무 쉬움** — 기존 eval_dataset.json 8건은 전부 단순 조회. 이를 발견하고 복잡 데이터셋(v2)을 만들어 테스트한 결과, 비교/추론 질문에서 실제 약점이 드러남 (아래 "복잡 데이터셋 테스트 결과" 참조)
2. **도구 자체의 정확성은 미검증** — Faithfulness는 "도구 결과 대비 충실성"이지 "도구 결과 자체가 맞는지"는 아님
3. **LLM 메트릭의 재현성** — 같은 답변을 두 번 평가하면 미세하게 다른 점수

### 복잡 데이터셋(v2) 테스트 결과

기존 단순 조회 8건 → 단순(3) + 비교(3) + 추론(2) + 엣지(2) = 10건으로 변경 후 테스트.

**1차 실행 (수정 전) — 겉보기 98점, 실제는 문제 투성이**

| 케이스 | 점수 | 실제 상태 |
|--------|------|-----------|
| 송파구 월세 | 100% | "처리 중 오류" — 에러인데 100점으로 판정 |
| 강남구 전세가율 | 100% | "처리 중 오류" — 도구 3회 제한에 걸림 |
| 매매 vs 전세 갈아타기 | 100% | "처리 중 오류" — 동일 |
| 판교 시세 | 100% | "처리 중 오류" — 지역코드 매핑 후 데이터 없음 |
| 지역 없이 "시세 알려줘" | 75% | 되물어야 하는데 임의로 3회 조회 |

발견된 문제:
1. **MAX_TOOL_CALLS=3이 복합 질문에 너무 적음** — 비교/추론은 최소 2개 도구 필요, 재시도하면 바로 3회 도달
2. **tool_limit_reached 후 에러 덮어쓰기** — 강제 응답 전송 후 스트림 소비 중 에러가 추가 전송되어 마지막 응답이 에러로 교체됨
3. **has_answer 체크 허술** — "처리 중 오류가 발생했습니다"를 유효 답변으로 판정 (길이>20자, `[ERROR]` 아님)
4. **LLM이 "지역 확인 요청" 규칙 무시** — 지역 없는 질문에 임의로 도구 호출

**수정 내용**:
- `MAX_TOOL_CALLS` 3→5 (프롬프트 + 코드 동기화)
- 프롬프트에 "비교 질문은 각 지역 한 번씩만", "지역 없으면 되묻기" 규칙 추가
- `tool_limit_reached` 상태에서 발생한 에러를 무시 (이미 응답 전송됨)
- `has_answer` 체크에서 에러/강제종료 메시지를 유효 답변에서 제외

**2차 실행 (수정 후) — 95점, 실제로도 개선**

| 카테고리 | 수정 전 | 수정 후 |
|----------|---------|---------|
| 단순 조회 | 100% (에러 포함) | 100% (진짜 정상) |
| 비교 분석 | 100% | 100% |
| 추론 필요 | 100% (에러 포함) | **75%** (1건 강제종료) |
| 엣지 케이스 | 88% | **100%** (되묻기 성공) |

**아직 남은 문제 — R01 "강남구 전세가율"**:
도구를 5회 호출했으나 강제 종료. 매매+전세 데이터를 각각 조회한 뒤 전세가율을 **계산**까지 해야 하는데, 데이터가 없는 월을 반복 탐색하면서 호출을 소진.

→ 이 문제를 해결하기 위해 **`calculate_jeonse_ratio` 도구를 신규 추가**함.

**3차 실행 (전세가율 도구 추가 후) — 추론 100%, 새로운 문제 발견**

| 카테고리 | 2차 | 3차 | 변화 |
|----------|-----|-----|------|
| 단순 조회 | 100% | 100% | 유지 |
| 비교 분석 | 100% | **75%** | C03 용산구 시점 비교 실패 |
| 추론 필요 | 75% | **100%** | R01 전세가율 1회 호출로 성공 |
| 엣지 케이스 | 100% | 88% | E02 LLM 일관성 부족 |

**성과**: `calculate_jeonse_ratio`가 매매+전세를 내부에서 한 번에 처리하므로, 기존 5회 호출 → **1회 호출**로 해결. R01, R02 모두 정상 응답.

**새로 발견된 구조적 문제 — "어느 시점까지 조회해야 하는가"**:

C03 "용산구 작년 이맘때랑 지금 비교해줘"가 5회 초과로 실패. 원인은 **거래 데이터의 시점 분포가 지역마다 다르기 때문**:

```
강남구: 매달 거래 수십 건 → 이번 달 조회하면 바로 데이터 있음
용산구: 거래가 뜸한 달이 있음 → 이번 달 없으면 1개월 전, 2개월 전... 반복 조회
어떤 시군구: 최근 거래가 5개월 전 → 5번 조회해도 못 찾을 수 있음
```

현재 로직은 "최근 1개월 조회 → 없으면 직전 1개월 추가"인데, 이걸 LLM이 판단하므로 거래가 드문 지역에서 호출을 낭비함.

→ **방안 1 "도구 내부 자동 탐색"을 적용**. 데이터 없으면 최대 6개월까지 자동으로 이전 월을 탐색하는 로직을 도구 안에 추가.

**4차 실행 (자동 탐색 적용 후)**

| 카테고리 | 3차 | 4차 | 변화 |
|----------|-----|-----|------|
| 단순 조회 | 100% | 100% | 유지 |
| 비교 분석 | 75% | **92%** | C03 용산구 시점 비교 **2회 호출로 성공** |
| 추론 필요 | 100% | 100% | 유지 |
| 엣지 케이스 | 88% | 62% | E01 판교 지역코드 매핑 실패 |

**핵심 성과**: C03 "용산구 작년 이맘때랑 지금 비교"가 해결됨. 작년 데이터가 해당 월에 없어도 도구가 자동으로 데이터 있는 월을 찾아 반환. LLM은 1회만 호출하면 됨.

```
수정 전: LLM이 "없네? 다른 월로" 반복 → 호출 5회 낭비 → 강제 종료
수정 후: 도구가 내부에서 6개월 탐색 → LLM은 1회 호출 → 정상 답변
```

데이터가 다른 월에서 발견되면 사용자에게 안내: "⚠️ 202603에는 데이터가 없어 202602 데이터를 조회했습니다."

**아직 남은 문제**:
- E01 "판교" — 자동 탐색이 아닌 **지역코드 매핑** 문제. "판교"→`_resolve_region_code()` 실패 → 도구가 에러 반환 → LLM이 재시도 반복. 폴백 딕셔너리에 동→시군구 매핑을 추가하거나, ES 지역코드 인덱스를 보강해야 함
- E02 "최근 아파트 시세" — LLM의 비결정적 특성. 같은 프롬프트인데 어떤 때는 되묻고 어떤 때는 임의 조회

## 4. 현재 도구(Tool) 목록

| 도구 | 용도 | 호출 예시 | 특징 |
|------|------|-----------|------|
| `search_apartment_trades` | 매매 실거래가 조회 | "강남구 매매가 알려줘" | 데이터 없으면 최대 6개월 자동 탐색, ES 캐시 + data.go.kr API |
| `search_apartment_rentals` | 전월세 실거래가 조회 | "송파구 전세 시세" | 동일한 자동 탐색 + 캐시 전략 |
| `calculate_jeonse_ratio` | 전세가율 계산 | "강남구 전세가율 알려줘" | 매매+전세를 내부에서 한 번에 조회, 전세가율·갭·판단근거 반환 |

### 도구 설계 원칙 (학습한 것)

1. **LLM은 "판단"만, 도구가 "실행"을 담당** — 데이터 탐색(월 변경), 계산(전세가율) 같은 반복/연산은 도구 내부에서 처리
2. **1회 호출로 완결** — 복합 질문(전세가율 = 매매 조회 + 전세 조회 + 계산)도 전용 도구로 만들면 LLM 호출 1회로 해결
3. **docstring이 도구 선택을 결정** — LLM은 docstring을 읽고 어떤 도구를 쓸지 판단하므로, 용도를 명확하게 작성
4. **자동 탐색 > LLM 재시도** — 데이터 없을 때 LLM이 월을 바꿔가며 재시도하면 호출 낭비, 도구가 알아서 찾아주는 것이 효율적

## 5. 실수와 개선

| 문제 | 원인 | 해결 | 교훈 |
|------|------|------|------|
| Tool 무한 루프 | LLM이 프롬프트 지시 무시 | 프롬프트 + 코드 이중 방어 | 중요한 제약은 코드로 강제 |
| 멀티턴 안 됨 | checkpointer 매번 새로 생성 | 한 번만 생성 후 재사용 | checkpointer 공유 = 대화 연속성 |
| 캐시가 옛 데이터 반환 | 1건만 있어도 캐시 히트 | TTL 기반 만료 추가 (6h/7d) | 신고 지연 데이터는 TTL 필수 |
| API 인증 실패 | 인코딩 키 사용 | 디코딩 키로 변경 | 공공 API는 반드시 직접 테스트 |
| 복합 질문 에러 (3회 제한) | MAX_TOOL_CALLS=3이 너무 적음 | 5로 상향 + 프롬프트 동기화 | 비교/추론은 도구 2개 이상 필요 |
| 에러가 응답 덮어쓰기 | tool_limit 후 스트림 에러 전파 | 이미 응답 전송 상태면 에러 무시 | 스트리밍에서 이벤트 순서가 중요 |
| 평가 메트릭 허점 | 에러 메시지를 유효 답변으로 판정 | has_answer에 에러 문구 필터 추가 | 평가 도구 자체도 검증해야 함 |
| 거래 드문 지역 조회 실패 | LLM이 월 변경하며 호출 낭비 | 도구 내부 자동 탐색 (최대 6개월) | "데이터 찾기"는 도구 책임, LLM은 판단만 |
| 전세가율 질문 호출 초과 | 매매+전세 각각 조회 → 호출 소진 | `calculate_jeonse_ratio` 도구 추가 | 복합 계산은 전용 도구로 1회 호출 해결 |

## 7. 학습 Q&A — 개념 이해도 체크 기록

잘 답한 것, 방향은 맞았지만 깊이가 부족했던 것, 몰라서 배운 것을 솔직하게 정리했습니다.

### 1주차 — 에이전트 만들기

#### Q1. ReAct에서 "강남구 전세가 알려줘"라고 물으면 어떻게 동작하는가?

**답변**: prompts.py에서 현재 날짜와 사용 가능한 도구를 확인 → 전세 질문이므로 `search_apartment_rentals` 호출 → 데이터 없으면 1~3개월 추가 조회 → 그래도 없으면 "데이터 없습니다" 안내. `create_agent()`에서 시스템 프롬프트를 전달하여 이 규칙이 적용됨.

**평가**: 정확. 추가로 이 "최대 3회 조회" 제한이 **이중으로** 걸려있다는 점을 보충함:
1. 프롬프트 레벨 — LLM이 자발적으로 따르도록 지시
2. 코드 레벨 — `MAX_TOOL_CALLS`로 강제 종료

LLM은 지시를 100% 따르지 않으므로, 중요한 제약은 프롬프트 + 코드 **벨트 앤 서스펜더** 전략으로 이중 방어해야 한다.

---

#### Q2. 왜 REST가 아니라 SSE 스트리밍인가?

**답변**: 답변 도중에도 변경이 일어날 수 있고, 서버에서 언제 끝날지 모르는 작업에 스레드를 점유하면 자원 낭비.

**평가**: 핵심을 잘 짚음. 보충하면:
- ReAct 루프는 `model`→`tools`→`model`→`done` 다단계 처리
- 일반 REST는 최종 답변 하나만 반환 가능 → 에이전트가 10초간 도구 3번 호출하는 동안 사용자는 빈 화면
- SSE는 각 단계를 실시간 전달 → "이해하고 있습니다" → "처리 중" → 최종 답변 순서로 UI 갱신

---

#### Q3. ToolStrategy vs 프롬프트로 "JSON으로 답해줘"의 차이?

**답변**: "자연어로 던진다 vs JSON 구조화로 던진다" 차이인지 잘 모르겠다.

**배운 것**: 방향은 맞았지만 핵심을 보충받음.

- **프롬프트만**: LLM이 자유 텍스트로 JSON을 생성 → 형식이 깨질 수 있음, 닫는 괄호 누락, 텍스트와 JSON 혼합 등
- **ToolStrategy**: LLM의 **도구 호출 메커니즘**을 활용 → ChatResponse를 "도구 호출"처럼 실행하므로 구조가 **100% 보장**

```
프롬프트만: "노력은 하지만 보장은 못하는" 상태
ToolStrategy: tool_calls[0]["args"]로 항상 정확한 dict 반환
```

`agent_service.py`에서 `tool.get("name") == "ChatResponse"`로 파싱하기 때문에 클래스명 변경 불가. 시스템 프롬프트의 Response Format 섹션은 "가이드", ToolStrategy는 "강제" — 둘이 동시에 존재하는 이유.

---

#### Q4. create_agent() 5가지 재료가 하나라도 빠지면?

**답변**:
- Model: 자연어 해석 자체가 불가
- Tools: 구체적 도구가 없어 모호한 결과
- System Prompt: 가이드라인 없이 너무 넓은 범위
- Response Format: 구조화 파싱 불가
- Checkpointer: 매번 정확한 프롬프트를 전달하지 않으면 답이 이상해짐

**평가**: 정확. Tools에 대해 보충 — "모호한 결과"보다 더 심각한 것은 **실제 데이터 조회 수단 자체가 없어** LLM 학습 데이터의 오래된/부정확한 정보로 답변(할루시네이션)한다는 것. 이 프로젝트 존재 이유가 "실거래가 데이터 기반 답변"이므로 tools 없는 에이전트는 일반 ChatGPT와 다를 바 없음.

### 2주차 Day 1 — Opik 트레이싱

#### Q5. Trace 분석만으로 부족한 이유, LLM Judge가 필요한 이유?

**답변**: 사람이 절대적 기준을 만들거나 많은 테스트 케이스를 만들 수 없기 때문에 LLM에게 자동 채점을 시킨다.

**평가**: 맞음. 추가로 **미시 vs 거시** 관점:

#### Q6. analyze_traces.py에서 Span을 조회하는 이유는? Trace만 보면 안 되나?

**답변**: 내부적으로 어떤 도구를 호출하고 각 구간마다 어떤 job을 호출했는지 보려고.

**평가**: 정확. 보충하면:
- **Trace** = 질문→답변 전체 기록 (입력, 최종 출력, 총 소요시간)
- **Span** = 그 안의 각 단계 (LLM 추론 → Tool 호출 → LLM 답변 생성)

Trace만 보면 "실패했다"는 알지만 **어디서 실패했는지** 모름. Span을 보면:
- Tool span 실패 → 지역코드 매핑 문제
- LLM span 이상 → 프롬프트 문제
- 특정 span 느림 → 병목 구간 특정

#### Q8. SSE 이벤트의 model/tools/done 세 단계가 ReAct 루프의 어디에 대응되나?

**답변**:
- model → 벡터화된 자연어를 LLM으로 던져 의도를 파악하는 단계
- tools → 의도를 파악하고 매칭되는 tools 어노테이션 메소드의 주석을 보고 판단하여 호출하는 단계
- done → 호출이 완료되고 SSE 형태로 데이터를 callback 하는 단계

**평가**: 방향은 맞았지만 각 단계의 경계가 섞여 있었음. 수정 포인트:

| step | ReAct 단계 | 실제로 일어나는 일 |
|------|-----------|-------------------|
| `model` | Reason + Act **결정** | LLM이 추론하고 "어떤 도구를 어떤 인자로 호출할지" **결정까지** 여기서 함 |
| `tools` | Act **실행** + Observe | 결정된 도구를 **실제로 실행** (data.go.kr API 호출) → 결과 수신 |
| `done` | 최종 Answer | 도구 결과를 바탕으로 LLM이 **사용자에게 줄 답변을 생성** (두 번째 LLM 호출) |

핵심: **도구 선택은 `model`에서 이미 완료**. `tools`는 선택이 아니라 실행. `done`은 콜백이 아니라 답변 생성.

#### Q19. GEval이란 무엇이고, ToolUsageMetric과 왜 둘 다 필요한가?

**답변**: 잘 모르겠다고 하여 설명받음.

**배운 것**:

#### Q21. 4가지 메트릭 중 하나만 봐도 되나? 왜 4개를 조합해야 하나?

**답변**: 잘 설명하지 못하여 설명받음.

**배운 것**: "송파구 전세 시세"에 매매 도구를 호출한 예시로:

| 메트릭 | 점수 | 판단 |
|--------|------|------|
| Faithfulness | **높음** | 매매 데이터를 정확히 옮겨 적음 |
| GEval + ToolUsage | **낮음** | 전세 질문에 매매 도구 → 도구 선택 오류 |
| AnswerRelevancy | 중간 | 그래서 답변도 부적절 |

Faithfulness만 보면 "잘했다" → 사실은 **틀린 답변**. 4개를 조합해야:
- **원인 분리**: GEval+ToolUsage 낮음 → "도구 선택이 잘못됨"
- **영향 파악**: AnswerRelevancy 중간 → "그래서 답변도 적절하지 않음"
- **착각 방지**: Faithfulness 높음 → "데이터는 정확히 옮겼지만 애초에 잘못된 데이터"

각 단계가 이전의 한계를 보완:
| 전환 | 보완한 한계 |
|------|-------------|
| Trace → Judge | 수동→자동, 개별→대량, 정성→정량(점수화) |
| Judge → DeepEval | 단일 관점→다각도, "좋다/나쁘다"→"어떤 측면이 왜 나쁜지" 분리 |

3단계를 합쳐도 남는 한계:
1. **평가 데이터셋이 곧 천장** — 데이터셋이 커버하지 못하는 질문 유형은 사각지대
2. **도구 자체의 정확성은 미검증** — Faithfulness는 "도구 결과 대비 충실성"이지, 도구가 반환한 데이터 자체가 맞는지는 아님
3. **LLM 메트릭의 재현성** — 같은 답변을 두 번 평가하면 미세하게 다른 점수

**답변**: 진단 — 사용자 입력에 맞춰 리턴을 받았는지, Judge — LLM이 제대로 검증했는지, DeepEval — 오픈소스 3가지가 정상 검증되었는지.

**평가**: 방향은 맞음. 코드 기준으로 보충:
- **진단**: 실패한 카테고리를 세서 많이 실패한 순 → `fail_cats.most_common(3)`
- **Judge**: 1~2점 답변을 찾아서 → `judge_score <= 2` → "저점수 답변 N건"
- **DeepEval**: 평균 0.7 미만인 메트릭 → `score < 0.7` → "Faithfulness 기준 미달"

각각 다른 관점의 "나쁜 것"을 잡아내서 합침:
- 진단 → **어떤 유형의 질문이 자주 실패하나?** (패턴)
- Judge → **어떤 답변이 품질이 낮나?** (개별 케이스)
- DeepEval → **어떤 측면이 기준 미달인가?** (메트릭)

#### Q23. 2주차 평가 체계 전체를 한 문장으로 요약하면? 아직 해결 못한 한계는?

**답변**: 사람과 컴퓨터(LLM)이 함께 검증을 수행하는 평가 프로젝트. 한계는 무엇을 만들 것이고 어떤 결과값을 기대하느냐에 따라 방향이 완전히 달라지므로 설계가 명확해야 한다. 그렇지 않으면 평가도 무의미.

**평가**: 핵심을 잘 짚음. 보충:

한 문장: **"에이전트가 왜 틀리는지를, 규칙(Day 2) → LLM(Day 3) → 다각도 메트릭(Day 4) → 종합(Day 5)으로 점점 정밀하게 진단하는 파이프라인"**

남은 한계:
| 한계 | 설명 |
|------|------|
| **설계 의존성** | "뭘 잘해야 하는가"가 정의 안 되면 메트릭도 데이터셋도 방향 없음 |
| **데이터셋이 곧 천장** | 사람이 만든 13건이 전부 → 커버 못하는 질문은 사각지대 |
| **평가도 LLM 의존** | Judge, DeepEval 모두 gpt-4o-mini가 채점 → 채점 자체의 정확성 검증 없음 |
| **실행 비용** | 13건 x 4메트릭 = 50+ LLM 호출 → 매번 돌리기엔 비용 부담 |

- 한국부동산원이 제공하는 건 **"부동산통계 조회 서비스"** (가격지수/변동률 등 통계 지표, 개별 단지 시세 아님)

| 소스 | 데이터 성격 | 비용 | 개별 단지 시세 |
|------|-------------|------|---------------|
| 실거래가 (현재 사용) | 과거 거래 신고가 | 무료 | 거래 없으면 없음 |

| KB시세 | 개별 단지 추정 시세 | 유료/비공개 | 가능하지만 접근 불가 |
