## 2. Day 1 — LangGraph StateGraph

### 핵심 개념: create_agent(ReAct) → StateGraph 전환

**ReAct (2주차)**:
- LLM이 "다음에 뭘 할지" 자유롭게 판단
- `tool_calls`가 있으면 프레임워크가 도구 실행 → LLM 다시 호출 (루프)
- 몇 번 돌지 예측 불가, LLM이 실수할 수 있음

**StateGraph (3주차)**:
- 코드가 노드(parse→fetch→respond)와 엣지(분기 조건)를 정의
- LLM은 각 노드에서 1회 호출, 흐름은 코드가 제어
- 호출 횟수 예측 가능, LLM 판단 오류 방지

### 구현한 StateGraph 구조

```
                    [START]
                      │
                      ▼
                 [parse_query]  ← LLM 1회: 질문 분석 (유형/지역/거래타입)
                      │
          ┌───────────┼───────────┐
          ▼           ▼           ▼
    "ambiguous"    "simple"    "compare"
          │           │           │
          ▼           ▼           ▼
  [ask_clarification] [fetch_simple] [fetch_compare]
          │           │           │
          ▼           └─────┬─────┘
        [END]               ▼
                        [respond]  ← LLM 1회: 답변 생성
                            │
                            ▼
                          [END]
```

### State (노드 간 전달되는 메모장)

```python
class AgentState(TypedDict):
    messages: list        # 대화 히스토리
    query_type: str       # "simple", "compare", "ambiguous", "jeonse_ratio"
    regions: list[str]    # ["강남구"] 또는 ["강남구", "송파구"]
    trade_type: str       # "매매", "전세", "전세가율"
    year_month: str       # 조회 년월
    data: dict            # 조회 결과
    response: str         # 최종 답변
```

### 5개 노드

| 노드 | 역할 | 실행 주체 |
|------|------|----------|
| `parse` | 질문 유형/지역/거래타입 분석 | LLM 1회 |
| `ask_clarification` | 모호한 질문에 되묻기 | LLM 1회 |
| `fetch_simple` | 단일 지역 데이터 조회 | 도구 (코드) |
| `fetch_compare` | 2개 지역 비교 조회 | 도구 (코드) |
| `respond` | 조회 데이터로 답변 생성 | LLM 1회 |

### 라우터 (분기 로직)

```python
def route_by_query_type(state):
    qt = state.get("query_type", "ambiguous")
    if qt == "compare":
        return "fetch_compare"
    elif qt in ("simple", "jeonse_ratio"):
        return "fetch_simple"
    else:  # ambiguous + 예상 못한 값 → 되묻기 (안전)
        return "ask_clarification"
```

### 2주차 대비 개선 효과

| 케이스 | 2주차 (ReAct) | 3주차 (StateGraph) |
|--------|-------------|-------------------|
| "판교 아파트 시세" | 5회 호출 → 강제 종료 | 0회 호출 → 즉시 되묻기 |
| "서울 매매 시세" | 강남구로 자의적 변환 | ambiguous → 되묻기 |
| "강남구 vs 송파구" | LLM이 1개만 조회할 수도 | 코드가 2회 호출 보장 |

### 핵심 이해 — ReAct vs StateGraph 차이

**"LLM이 루프를 돌리는가, 코드가 루프를 돌리는가"**

- ReAct: LLM 응답에 `tool_calls`가 있으면 프레임워크가 도구 실행 후 LLM 재호출 (LLM이 트리거)
- StateGraph: `add_edge`와 `add_conditional_edges`가 다음 노드를 강제 (코드가 트리거)
- LLM 자체는 항상 "1번 호출 → 1번 응답". 여러 번 호출하는 건 프레임워크가 하는 것

### SSE 연동 (GraphAgentService)

- `graph_agent_service.py`: StateGraph 스트리밍을 기존 SSE 형식(model→tools→done)에 변환
- `chat.py`: `AGENT_MODE` 환경변수로 graph/react 전환 (기본: graph)
- 기존 `agent_service.py` 유지 — `AGENT_MODE=react`로 복원 가능

### 미해결

- 멀티턴 대화 안 됨 (initial_state에서 이전 대화를 로드하지 않음 → Day 4에서 수정)
- 시점 비교("작년 이맘때 vs 지금") 분기 미구현

## 3. Day 2 — PDF RAG 파이프라인

### RAG = Retrieval + Augmented + Generation

```
Day 2 (완료): PDF → 파싱 → 청킹 → 임베딩 → ES 적재    ← Retrieval 준비
Day 3 (예정): 질문 → ES 검색 → LLM에 전달 → 답변 생성  ← 실제 RAG
```

### 파이프라인 4단계

```
PDF 파일 (원본)
  ↓ Step 1: 파싱 (pdf_loader.py)
  ↓   PyPDFLoader로 페이지별 텍스트 추출
  ↓
페이지별 Document (9페이지 → 9개 Document)
  ↓ Step 2: 청킹 (chunker.py)
  ↓   RecursiveCharacterTextSplitter (500자, 100자 겹침)
  ↓   문맥 유지되도록 문장/단락 단위로 분할
  ↓
청크 리스트 (9페이지 → 약 30청크)
  ↓ Step 3: 임베딩 (embedder.py)
  ↓   OpenAI text-embedding-3-small → 1536차원 벡터
  ↓   "서울 아파트 가격"이라는 의미가 숫자 벡터로 변환
  ↓
벡터 + 텍스트 + 메타데이터
  ↓ Step 4: ES 적재 (es_client.py)
  ↓   edu-real-estate-reports 인덱스에 벌크 저장
  ↓
ES에 저장된 최종 구조:
  content: "2025년 주택시장 하향 안정..."     ← BM25 키워드 검색용
  content_vector: [0.023, -0.041, ...]        ← kNN 벡터 검색용
  metadata: {source_file: "kb_...", page: 3}  ← 출처 추적용
```

### 검색 방식 3가지

| 방식 | 원리 | 장점 |
|------|------|------|
| BM25 | "강남구", "전망" 키워드 매칭 | 정확한 키워드 포함 문서 |
| Vector (kNN) | 질문 벡터와 청크 벡터 코사인 유사도 | 의미적으로 유사한 문서 |
| **Hybrid** | BM25 + kNN 점수 합산 | 둘의 장점 결합 |

**RRF(Reciprocal Rank Fusion)는 ES Platinum 라이선스 필요** → score 합산 방식으로 대체.

|------|------------|-----|

## 4. Day 3 — 하이브리드 검색 + StateGraph 연동

### comprehensive 경로 추가

기존 simple/compare/ambiguous 분기에 **comprehensive** 경로를 추가하여, PDF 보고서 기반 종합 분석이 가능해짐.

```
parse → fetch → search_pdf → respond
```

- `parse_query`에서 "전망", "투자", "분석" 등 키워드가 포함되면 `comprehensive`로 분류
- `fetch` 노드에서 실거래가 API를 먼저 조회한 뒤, `search_pdf` 노드에서 ES 하이브리드 검색으로 PDF 보고서 청크를 가져옴
- `respond` 노드에서 실거래가 수치 + PDF 전문가 분석을 종합하여 답변 생성

### 하이브리드 검색 (BM25 + kNN) StateGraph 연동

- Day 2에서 구축한 BM25 + kNN 하이브리드 검색을 StateGraph의 `search_pdf` 노드로 통합
- 질문 벡터와 PDF 청크 벡터의 코사인 유사도(kNN) + 키워드 매칭(BM25) 점수 합산
- 상위 5개 청크를 `state["pdf_context"]`에 저장하여 `respond` 노드에서 참조

### 실거래가 요약본 전달 (토큰 절약)

- comprehensive 경로에서는 실거래가 전체 데이터 대신 **요약 통계(평균가/최고가/최저가/거래건수)**만 `respond` 노드에 전달
- PDF 청크가 이미 상당한 토큰을 차지하므로, 실거래가는 요약으로 토큰 절약

### 답변 서식 개선

프롬프트에 답변 서식 가이드라인을 추가하여 가독성 향상:

- 이모지 섹션 헤더 (예: 📊 시세 요약, 📈 전망)
- 구분선(---) 으로 섹션 분리
- ▸ 수치 데이터 인라인 표기
- ✅ 결론 섹션으로 핵심 요약

### CancelledError 원인 분석

- SSE 스트리밍 중 브라우저가 연결을 끊으면(탭 닫기, 새 질문 입력 등) 서버에서 `asyncio.CancelledError` 발생
- FastAPI의 async generator가 중단되면서 LangGraph의 `astream()`도 취소됨
- 해결: try/except로 `CancelledError`를 잡아 graceful 종료 처리

### react-markdown 시도 → 프롬프트 서식으로 전환

- 프론트엔드에서 `react-markdown`으로 마크다운 렌더링 시도
- 기존 CSS와 충돌하여 레이아웃 깨짐 발생
- 마크다운 렌더링 대신 **프롬프트 서식**(이모지 + 구분선 + 기호)으로 전환하여 plain text로도 가독성 확보

## 5. Day 4 — AsyncSqliteSaver + 멀티턴 대화

### AgentState에 add_messages reducer 추가

```python
class AgentState(TypedDict):
    messages: Annotated[list, add_messages]  # reducer 추가
    query_type: str
    regions: list[str]
    trade_type: str
    year_month: str
    data: dict
    response: str
```

- `add_messages` reducer를 사용하면 노드가 반환하는 메시지가 기존 리스트에 **append** 됨
- reducer 없이는 매번 덮어쓰기 → 이전 대화가 사라지는 문제 발생
- 이 변경으로 멀티턴 대화의 기본 전제 확보

### InMemorySaver → AsyncSqliteSaver 전환 (영속 저장)

- `InMemorySaver`: 서버 재시작 시 모든 대화 히스토리 소실
- `AsyncSqliteSaver`: SQLite 파일에 체크포인트 영속 저장
- 서버를 재시작해도 이전 대화 맥락이 유지됨
- `thread_id` 기반으로 세션별 대화 분리

### trim_messages 적용 (토큰 폭증 방지)

- 멀티턴 대화가 길어지면 전체 히스토리를 매번 LLM에 전송 → 토큰 비용 가속 증가
- `trim_messages(max_tokens=4000)` 적용: 오래된 메시지부터 잘라냄
- `MAX_MESSAGE_TOKENS=4000` — 시스템 프롬프트 + 현재 질문 토큰을 고려한 값

### parse_query에 이전 대화 맥락 포함

- "거기서 전세가도 알려줘", "아까 본 지역에서 비교해줘" 같은 후속 질문 처리
- `parse_query` 노드에서 `state["messages"]`의 최근 대화를 참조하여 지역/거래타입 추론
- 이전 대화에서 "강남구 매매"를 조회했다면, "거기서 전세가도" → 강남구 전세로 해석

### 15턴 테스트 전부 통과

다양한 시나리오로 15턴 연속 테스트 수행, 전부 통과:

| 시나리오 | 테스트 내용 | 결과 |
|----------|------------|------|
| 맥락 유지 | "강남구 매매" → "거기서 전세가도" | ✅ 강남구 전세 조회 |
| 지역 전환 | "강남구 매매" → "송파구는?" | ✅ 송파구 매매 조회 |
| 되묻기 | "판교 아파트 시세" | ✅ 분당구로 되묻기 |
| 종합 판단 | "강남구 투자 전망" | ✅ 실거래가 + PDF 종합 |
| 동→시군구 | "역삼동 매매가" | ✅ 강남구로 안내 |
| 비교 후속 | "강남 vs 송파" → "거기서 전세가율도" | ✅ 두 지역 전세가율 비교 |

### MCP 토큰 테스트

- 다른 세션에서 MCP(Model Context Protocol) 토큰 사용량 테스트도 완료
- 멀티턴 대화 시 토큰 소비 패턴 확인 → `trim_messages`의 효과 검증

## 6. Day 5 — ReAct 메인 + StateGraph 서브에이전트 전환

### 최종 아키텍처 결정

3주차 Day 1~4에서 StateGraph 전체 에이전트(`real_estate_graph.py`)를 구축했지만, 최종적으로 **ReAct 메인 + StateGraph 서브에이전트** 구조로 전환.

- **메인 에이전트**: `real_estate_agent.py`의 ReAct (`create_agent()` + `ToolStrategy(ChatResponse)`)
- **서브에이전트**: 같은 파일 내 StateGraph로 구현한 PDF 하이브리드 검색(`search_pdf_reports` 도구)
- **`AGENT_MODE` 기본값**: `react` (chat.py에서 `os.getenv("AGENT_MODE", "react")`)

### 구조: ReAct + StateGraph 서브에이전트

```
[ReAct 에이전트] ← create_agent() + ToolStrategy(ChatResponse)
    │
    ├─ search_apartment_trades      ← @tool (data.go.kr API)
    ├─ search_apartment_rentals     ← @tool (data.go.kr API)
    ├─ calculate_jeonse_ratio       ← @tool (내부 계산)
    └─ search_pdf_reports           ← @tool (StateGraph 서브에이전트)
         │
         ├─ [BM25 검색] ──┐
         ├─ [Vector 검색] ─┤ (병렬 fan-out)
         ├─ [merge] ───────┘ (fan-in)
         └─ [format] → 결과 반환
```

### 전환 이유

| 관점 | StateGraph 전체 | ReAct 메인 + StateGraph 서브 |
|------|----------------|----------------------------|
| 흐름 제어 | 코드가 완전 제어 (예측 가능) | LLM이 도구 선택 (유연) |
| PDF 검색 | `search_pdf` 노드 (단일 검색) | `search_pdf_reports` 도구 (BM25+Vector 병렬) |
| 확장성 | 새 경로 추가 시 노드+엣지 필요 | 새 도구만 추가하면 됨 |
| SSE 호환 | `graph_agent_service.py` 별도 구현 | 기존 `agent_service.py` 그대로 사용 |

### StateGraph 전체 에이전트 유지

`real_estate_graph.py`와 `graph_agent_service.py`는 삭제하지 않고 `AGENT_MODE=graph`로 전환 가능하도록 유지. intent 분기, 멀티턴 trim_messages, AsyncSqliteSaver 등 graph 모드만의 장점이 있음.

## 7. 개념 학습 기록

### LLM 토큰화 → 임베딩 → 어텐션

**토큰화 (BPE)**: 텍스트를 조각(토큰)으로 분리. 자주 붙어 나오는 글자 조합을 하나로 합치는 것을 반복 → 약 10만개 토큰 사전 생성.

**임베딩**: 토큰 ID → 의미를 담은 숫자 벡터 (1536~4096차원). 비슷한 의미의 단어는 비슷한 벡터. 학습 과정에서 자동으로 결정됨.

**어텐션**: "이 단어를 이해하려면 어떤 단어에 집중해야 하는가?" 모든 토큰 쌍의 관련도를 계산 (N² 비용). 대화가 길어지면 토큰 증가 → 연산 폭증 → 비용 증가.

### 멀티턴 대화와 토큰 비용

- InMemorySaver: 메모리에 대화 저장 (서버 재시작 시 소실)
- AsyncSqliteSaver: 파일에 영속 저장 (Day 4에서 전환)
- **매 턴마다 전체 대화 히스토리를 LLM에 다시 보냄** → 대화 길어지면 비용 가속 증가
- 해결: 윈도우 방식 (최근 N턴만) 또는 요약 방식 (오래된 대화 요약)

## 8. 학습 Q&A

### Day 1 — StateGraph

**Q1. add_edge vs add_conditional_edges 차이**
- 내 답변: 잘 모르겠음
- 배운 것: `add_edge` = 무조건 이동 (직선 도로), `add_conditional_edges` = 분기 함수 결과에 따라 다른 노드로 (갈림길)

### Day 2 — PDF 파이프라인

**Q4. 청크 크기 500자, 겹침 100자 설정 이유**
- 내 답변: 청크를 너무 늘려도 토큰 소비가 늘고 줄여도 환각 발생. 겹치는 이유는 대화와 문장의 맥락 파악을 위함
- 평가: ✅ 정확

### Day 3 — 하이브리드 검색 + 서식 개선

**Q3. 리랭킹과 cross-encoder vs bi-encoder 차이**
- 내 답변: 리랭킹은 자연어 매칭과 벡터 매칭 점수를 비교해서 높은 것으로 유도하는 것. cross-encoder는 잘 모르겠음
- 배운 것: bi-encoder는 query와 document를 따로 벡터화 후 거리 비교 (빠르지만 대략적), cross-encoder는 (query+document)를 함께 읽고 관련도 점수 (느리지만 정밀). 해시 비유: 내부 알고리즘 몰라도 결과(점수)를 활용하면 됨

**Q5. 서브에이전트 전체 검색 흐름**
- 내 답변: 잘 모르겠음
- 배운 것: 호출 → BM25(병렬) + Vector(병렬) → merge(중복제거) → rerank(cross-encoder 재정렬) → format(텍스트 변환) → 반환. 도서관 비유: 사서 A(BM25)와 사서 B(Vector)가 동시에 찾고, 전문가(rerank)가 직접 읽고 판단

**Q6. comprehensive 경로에서 실거래가를 요약본만 전달하는 이유**
- 내 답변: 이미 메모리에 값을 가지고 있기 때문?
- 배운 것: 토큰 절약이 목적. PDF 결과가 이미 토큰을 많이 차지하니 실거래가는 요약본(통계+상위3건)만 전달. 99건 전부 보내도 LLM은 결국 요약해서 답하니 품질 동일

### Day 4 — 멀티턴 대화 + 영속 저장

**Q2. "자세히 보여줘"가 StateGraph에서 안 되고 ReAct에서 되는 이유**
- 내 답변: 랭그래프는 정해진 길만 따라가고 완벽한 분기처리가 없으면 안 되는데, ReAct는 LLM이 판단에 맞춰 직접 답변할지 도구를 쓸지 판단하기 때문
- 평가: ✅ 정확

**Q7. 최종 아키텍처에서 ReAct와 StateGraph 각각의 역할**
- 내 답변: ReAct는 LLM이 판단하여 어떻게든 답을 내기 위한 장치, Tool이 필요하면 호출. StateGraph는 Tool 안에서 규칙이나 병렬처리가 필요할 때 사용
- 평가: ✅ 정확. "ReAct = 무엇을 할지 (유연), StateGraph = 어떻게 할지 (정확)"
