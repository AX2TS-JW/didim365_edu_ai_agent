### Deep Agent 5대 기능 실사용 분석

Opik trace 분석 결과, Deep Agent SDK의 핵심 기능 중 실제로 활용되지 않는 것들이 확인됨:

| # | 기능 | 코드 존재 | 실제 사용 | 비고 |
|---|------|----------|----------|------|
| 1 | write_todos (계획 분해) | ✅ SDK 내장 | ❌ Opik 미확인 | LLM이 직접 호출 경로 선택 |
| 2 | VFS (write_file/read_file) | ✅ 프롬프트 가이드 | ❌ Opik 미확인 | 도구 재호출로 해결 |
| 3 | task() 서브에이전트 위임 | ✅ 3개 정의 | ❌ Opik 미확인 | 메인이 직접 도구 반복 호출 |
| 4 | 장기 기억 (cross-thread) | ❌ 미구현 | ❌ | AGENTS.md는 정적 파일 로드 (read-only) |
| 5 | CompositeBackend | ❌ 미구현 | ❌ | 기본 StateBackend만 사용 |

**원인**: 하이브리드 B구조(tools=[4개])에서 LLM이 직접 도구 호출로 충분히 답변을 만들어내므로, 굳이 TODO 분해 → task() 위임 → VFS 공유라는 복잡한 경로를 선택하지 않음. 프롬프트에 가이드는 있지만 강제하지 않으면 LLM은 단순한 경로를 선호.

**개선 방향** (미적용):
- 프롬프트 강화: "3개 이상 지역 비교 시 반드시 task() 사용" 등 강제 규칙
- CompositeBackend + StoreBackend 구현으로 진짜 장기 기억 적용
- A구조(tools=[])로 전환하면 task() 사용이 강제되지만 단순 질문 성능 저하

## 8. 학습 Q&A

### Day 1 — Deep Agent SDK

**Q1. create_agent()와 create_deep_agent()의 차이**
- 내 답변: ReAct 하나로 단일 선형 작업과, 분기가 나와서 독립적으로 동작한다의 차이
- 평가: ⚠️ 방향은 맞지만 깊이 부족. 서브에이전트 위임 외에도 SDK 미들웨어 개입으로 recursion 소모 증가, 내장 도구(VFS, TODO) 자동 추가 등이 핵심 차이
- 배운 것: 반환값은 동일(CompiledStateGraph)하지만, create_deep_agent()는 SDK 내장 미들웨어(TodoListMiddleware, MemoryMiddleware, PatchToolCallsMiddleware)가 매 스텝마다 개입. 내장 도구(write_todos, VFS, task())도 자동 추가됨. 결과적으로 도구 호출 1회당 recursion 소모가 ReAct(2~3스텝) 대비 Deep Agent(3~4스텝)로 증가

**Q2. recursion_limit과 MAX_TOOL_CALLS의 관계**
- 내 답변: recursion은 Deep Agent의 최대 재귀호출 제한, MAX는 전체적인 LLM 호출의 최대 제한 횟수
- 평가: ⚠️ 방향은 맞지만 부정확. recursion_limit은 LLM 호출이 아니라 그래프 노드 실행(미들웨어 포함) 횟수. MAX_TOOL_CALLS도 LLM 호출이 아니라 도구 실행 횟수
- 배운 것: 두 제한은 서로 다른 계층에서 동작. MAX_TOOL_CALLS = 사용자 관점의 도구 호출 횟수 제한, recursion_limit = 시스템 관점의 그래프 노드 실행 횟수 제한. `recursion_limit ≥ MAX_TOOL_CALLS × 4`를 지켜야 MAX_TOOL_CALLS가 먼저 동작하여 정상 종료. 독립적으로 설정하면 recursion_limit에 먼저 걸려 GraphRecursionError(비정상 종료) 발생

### Day 2 — VFS/TODO

**Q3. VFS(Virtual File System)가 실제 파일 시스템과 다른 점**
- 내 답변: 잘 모르겠음
- 평가: 몰라서 배움
- 배운 것: VFS는 state["files"] 딕셔너리에 저장되는 인메모리 가상 파일. 디스크에 쓰지 않고 그래프 상태 안에서만 존재. 서브에이전트 간 데이터 공유(DataCollector → write_file → Analyst → read_file)에 사용되며, 도구 재호출 없이 데이터를 재활용하여 API 비용 절약

**Q4. write_todos의 역할과 context rot 방지**
- 내 답변: 잘 모르겠음
- 평가: 몰라서 배움
- 배운 것: 복합 질문을 단계별로 분해하여 작업 목록을 만듦. LLM은 대화가 길어지면 초기 맥락을 잊는데(context rot), TodoListMiddleware가 매 스텝마다 TODO 상태를 체크하여 "다음에 뭘 해야 하는지"를 상기시켜줌

### Day 3 — 서브에이전트

**Q5. A구조(서브만)와 B구조(하이브리드)의 차이와 선택 이유**
- 내 답변: 서브만 있는 건 모든 작업이 서브로 타는 것, 하이브리드는 기존 tools 유지하면서 서브도 쓰는 것. B 선택 이유는 간단한 작업도 A를 타서 느렸기 때문
- 평가: ✅ 정확
- 배운 것: A구조(tools=[])는 모든 질문을 task()로 서브에이전트에 위임 → 단순 질문도 서브에이전트 생성 오버헤드(10초). B구조(tools=[4개])는 메인이 직접 도구 호출 가능 → 단순 질문은 빠르게(6초), 복합 질문만 task() 위임. LLM이 질문 복잡도에 따라 자연스럽게 경로를 선택함

**Q6. 서브에이전트와 @tool의 근본적 차이**
- 내 답변: 서브에이전트는 독립된 실행공간에서 연속적으로 동작하는 tool, @tool은 단일 선형의 작업
- 평가: ✅ 정확
- 배운 것: @tool은 함수 1개 = 기능 1개, 자체 판단 없음, 메인 컨텍스트 공유. 서브에이전트는 자체 LLM을 가진 독립 에이전트, 여러 도구를 조합하여 스스로 판단, 컨텍스트 격리. "직원에게 심부름 시키기(@tool) vs 전문가에게 프로젝트 위임(서브에이전트)"

### Day 4 — 장기 기억

**Q7. AGENTS.md가 시스템 프롬프트와 다른 점**
- 내 답변: 잘 모르겠음
- 평가: 몰라서 배움
- 배운 것: 시스템 프롬프트는 코드에 하드코딩된 행동 규칙(도구 선택 기준, 응답 형식 등). AGENTS.md는 외부 파일로 관리되는 도메인 지식(전세가율 기준, 지역 특성 등). AGENTS.md는 코드 변경 없이 지식을 업데이트할 수 있고, MemoryMiddleware가 자동으로 에이전트에 로드

### Day 5 — 종합 테스트 + 트러블슈팅

**Q8. GeneratorExit가 발생하는 원인과 해결**
- 내 답변: AI 외의 기술이라 잘 모르겠음
- 평가: 몰라서 배움 (Python 비동기 + SSE 스트리밍 영역)
- 배운 것: 프론트엔드가 done 이벤트 수신 후 controller.abort()로 SSE 연결을 끊으면, 백엔드의 LangGraph astream() async generator가 끝까지 소비되지 않고 GC됨 → GeneratorExit 발생 → Opik에 에러로 기록. 해결: ChatResponse yield 후 asyncio.ensure_future()로 남은 stream을 background에서 drain. SSE 연결 끊김과 무관하게 LangGraph가 정상 종료됨. 비유: 상대방이 전화 끊었는데 아직 할 말이 남아서 에러 → 남은 말을 혼잣말로 끝까지 하고 정상 종료

**Q9. AGENT_MODE 기본값 불일치 버그가 위험한 이유**
- 내 답변: 재귀 호출 구조에서 서로 안 맞으면 응답 내기 전에 cut 해버리는 문제
- 평가: ⚠️ 방향은 맞지만 핵심이 다름. 재귀 제한 문제가 아니라 "모드 자체가 달라지는" 문제
- 배운 것: chat.py는 기본값 "deep", agent_service.py는 기본값 "react"로 달랐음. .env에 AGENT_MODE를 안 넣으면 chat.py는 Deep Agent라고 판단하여 AgentService를 사용하지만, agent_service.py는 ReAct 에이전트를 생성하고 MAX_TOOL_CALLS=5로 설정. Deep Agent라고 생각했는데 실제로는 ReAct가 5회 제한으로 동작. 환경변수 기본값은 한 곳에서 관리해야 함

**Q10. 체크포인트 정합성 문제와 패치 방법**
- 내 답변: 잘 모르겠음
- 평가: 몰라서 배움
- 배운 것: MAX_TOOL_CALLS 강제 종료 시 체크포인트에 "미완료 tool_calls"가 남음. 다음 요청에서 이전 상태를 로드하면 tool_calls에 대한 ToolMessage가 없어서 OpenAI 400 에러 발생. 해결: aget_state()로 상태 확인 → 미완료 tool_calls 발견 시 더미 ToolMessage 패치 → aupdate_state()로 상태 복구. 비유: "질문은 했는데 답을 못 받은 상태"를 "답 없음"으로 채워넣어 정합성을 맞추는 것

## 9. 1~4주차 전체 흐름

```
1주차: "강남구 매매가" → 단일 Tool + ES
2주차: "왜 틀렸지?" → Opik + DeepEval 평가
3주차: "비교해줘" → StateGraph + PDF RAG + 멀티턴 + ReAct+서브에이전트
4주차: "투자 리포트" → Deep Agent + 서브에이전트 3개 + VFS + 장기기억
```
