## 2. Day 1 — Deep Agent SDK 적용

### create_agent() → create_deep_agent() 교체

```python
# 3주차
agent = create_agent(model, tools=[...], system_prompt=..., response_format=...)

# 4주차
agent = create_deep_agent(model, tools=[...], system_prompt=..., response_format=..., subagents=[...], memory=[...])
```

- 반환값 동일 (CompiledStateGraph) → 기존 astream() 호환
- SDK가 자동으로 내장 도구 추가: write_todos, read_todos, write_file, read_file, ls, glob, grep, task()
- 기존 도구 4개(trades, rentals, ratio, pdf) 그대로 사용

## 3. Day 2 — VFS/TODO 활용

### TODO (write_todos)
- 복합 질문("5개구 비교 리포트")을 단계별로 분해
- LLM이 작업 진행 상태를 추적 → context rot 방지

### VFS (write_file/read_file)
- state["files"] 딕셔너리에 가상 파일 저장 (실제 디스크 파일 아님)
- 서브에이전트 간 데이터 공유: DataCollector → write_file → Analyst → read_file
- 도구 재호출 없이 데이터 재활용 → API 비용 절약

### 실제 동작
- 단순 질문: VFS 안 쓰고 도구 직접 호출 (정상)
- 복합 질문: write_todos로 분해 후 task()로 서브에이전트 위임

## 4. Day 3 — 서브에이전트

### A구조 vs B구조 (하이브리드) 결정 과정

| 구조 | 메인 도구 | 서브에이전트 | 결과 |
|------|----------|------------|------|
| A (서브만) | tools=[] | task()로만 | 단순 질문도 느림 (10초) |
| B (하이브리드) | tools=[4개] | 복합만 task() | 단순 빠름(6초) + 복합은 task() |

**B구조(하이브리드) 채택**: 단순 질문은 직접, 복합 질문만 서브에이전트 위임.

### 서브에이전트 3개

| 이름 | 역할 | 도구 |
|------|------|------|
| data-collector | 데이터 수집 + VFS 저장 | trades, rentals, pdf |
| analyst | 비교 분석 + VFS 읽기/저장 | ratio, trades, rentals |
| reporter | 종합 리포트 작성 | 없음 (LLM만) |

### 서브에이전트 vs @tool 차이

| | @tool (3주차) | 서브에이전트 (4주차) |
|---|-------------|-------------------|
| 자체 판단 | 없음 | 있음 (자체 LLM) |
| 컨텍스트 | 메인과 공유 | 격리 (독립) |
| 복잡도 | 단일 기능 | 여러 도구 조합 |

### 알려진 한계
