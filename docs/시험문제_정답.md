### Q1. `@tool` 데코레이터 적용 위치 — 정답: **(B)**

```python
@tool
def search_hospital(region: str) -> str:
    """병원을 검색합니다."""
    return f"{region} 병원 목록"
```

**해설:** `@tool`은 **독립 함수(standalone function)** 에만 사용해야 한다.

| 보기 | 판정 | 이유 |
|------|------|------|
| (A) | ✗ | 클래스 메서드에 `@tool`을 붙이면 `self`가 tool 스키마에 노출됨 |
| **(B)** | **✓** | 독립 함수 + docstring + 타입 힌트 → 올바른 사용법 |
| (C) | ✗ | `tool(search_hospital)` 팩토리 호출은 가능하지만, 데코레이터 패턴이 공식 권장 |
| (D) | △ | `__init__` 내부 클로저로 우회 가능하나, 공식 문서 권장 패턴이 아님 |

**프로젝트 근거:** `app/agents/tools.py`에서 `@tool`을 독립 함수에 적용하고 있음.

### Q2. Tool docstring의 역할 — 정답: **(B)**

> LLM이 어떤 Tool을 사용할지 결정하는 근거(메타데이터)로 사용된다.

**해설:** `@tool` 데코레이터는 함수의 docstring을 `description` 필드로 추출하여 LLM에게 전달한다. LLM은 이 description을 읽고 "이 tool을 호출할지 말지"를 판단한다. 따라서 docstring이 부실하면 LLM이 적절한 상황에 tool을 선택하지 못하거나 잘못된 tool을 호출할 수 있다.

**프로젝트 근거:** `tools.py`의 `search_apartment_trades`와 `search_apartment_rentals`에 상세한 docstring이 작성되어 있으며, 이를 통해 LLM이 매매/전월세를 구분하여 호출한다.

### Q3. Checkpointer 싱글턴 관리 — 정답: **(B)**

> 코드 B — `MemorySaver()`를 매 호출마다 생성하면 `thread_id` 기반 대화 연속성이 파괴된다.

**해설:**
- **코드 A (정상):** 모듈 레벨 싱글턴이므로 모든 요청이 동일한 `MemorySaver` 인스턴스를 공유 → `thread_id` 기반 대화 이력 유지됨
- **코드 B (버그):** 함수 호출마다 새 `MemorySaver()`를 생성 → 매번 빈 상태로 시작 → 이전 대화를 기억하지 못함

(A)의 우려("여러 요청이 메모리를 공유해 버그")는 틀린 설명이다. `thread_id`로 격리되므로 공유가 오히려 **필요한** 동작이다.

**프로젝트 근거:** 실제로 이 버그를 겪었음. `chat.py`에서 `AgentService()`를 요청마다 생성하여 멀티턴 대화가 안 되는 문제 발생 → 모듈 레벨 싱글턴으로 수정하여 해결 (커밋 `ac3b0fc`).

### Q4. 올바른 import 경로 — 정답: **(B), (E)**

| 보기 | 판정 | 설명 |
|------|------|------|
| (A) `from langchain.tools import tool` | △ | 동작하지만 `langchain_core`에서 re-export된 것. 직접 사용 권장하지 않음 |
| **(B)** `from langchain_core.tools import tool` | **✓** | 정식 canonical 경로 |
| (C) `from langchain.agents.structured_output import ToolStrategy` | △ | 동작하며 프로젝트에서도 사용 중이나, 공식 문서에서 `langchain_core` 우선 권장 |

### Q6. `response_format`과 `ToolStrategy` — 정답: **(B)**

> 방법 B가 올바르다. `response_format=ToolStrategy(ChatResponse)`가 공식 API다.

**해설:**
- **방법 A의 문제:** `ChatResponse`를 `@tool`로 정의하면, LLM이 이를 "일반 도구" 중 하나로 인식한다. 반드시 호출한다는 보장이 없고, 다른 tool과 혼동될 수 있다.
- **방법 B (정답):** `response_format=ToolStrategy(ChatResponse)`는 LLM의 **최종 응답 형식**을 강제한다. 에이전트가 작업을 마치면 반드시 `ChatResponse` 스키마에 맞춰 응답해야 한다.

**프로젝트 근거:** `real_estate_agent.py:63`에서 정확히 방법 B를 사용 중:
```python
response_format=ToolStrategy(ChatResponse)
```

### Q7. 클래스 메서드와 `@tool` — 정답: **(B)**

> `self` 파라미터가 LangChain tool 스키마에 노출되어 LLM이 `self`를 인자로 채우려 시도한다.

**해설:** `@tool` 데코레이터는 함수 시그니처를 분석하여 LLM에게 전달할 파라미터 스키마를 생성한다. 클래스 메서드의 `self`도 일반 파라미터로 인식되어 스키마에 포함된다. LLM은 `self`에 어떤 값을 넣어야 할지 알 수 없으므로, 엉뚱한 값을 채우거나 에러가 발생한다.

**해결 방법:** 독립 함수로 분리하거나, 클로저 패턴으로 `self`를 캡처한다.

### Q8. Checkpointer와 멀티턴 대화 — 정답: **(B)**

> `create_agent()`에 `checkpointer` 파라미터로 `MemorySaver()` 등 전달 + 각 invoke에 `{"configurable": {"thread_id": "..."}}` 전달

**해설:** 멀티턴 대화에는 **두 가지**가 모두 필요하다:
1. **Checkpointer:** 대화 상태를 저장할 저장소 (예: `InMemorySaver()`)
2. **thread_id:** 어떤 대화 스레드인지 식별하는 키

`checkpointer`만 있고 `thread_id`가 없으면 어떤 대화인지 구분할 수 없고, `thread_id`만 있고 `checkpointer`가 없으면 저장 자체가 안 된다.

**프로젝트 근거:**
- `real_estate_agent.py:64` → `checkpointer=checkpointer`
- `agent_service.py` → `{"configurable": {"thread_id": thread_id}}`

### Q9. AI Hallucination 식별 — 정답: 해당 없음 (모두 정상 동작)

| 보기 | 판정 | 설명 |
|------|------|------|
| (A) `from langchain.agents.structured_output import ToolStrategy` | ✓ 정상 | 프로젝트에서 실제 사용 중 (`real_estate_agent.py:13`) |
| (B) `from langchain_core.tools import tool` | ✓ 정상 | 공식 canonical 경로 |
| (C) `from langchain.agents import create_agent` | ✓ 정상 | 프로젝트에서 실제 사용 중 (`real_estate_agent.py:12`) |
| (D) `from langchain_core.messages import HumanMessage` | ✓ 정상 | 공식 canonical 경로 |
| (E) `response_fromat = ToolStrategy(ChatResponse)` | ✓ 정상 | 변수명 오타(`fromat`)이지만, `response_format=response_fromat`으로 올바른 키워드 인자에 전달되므로 실행 오류 없음 |

**해설:** 이 문제의 핵심은 "AI가 존재하지 않는 import 경로를 그럴듯하게 만들어내는 현상(hallucination)"을 식별하는 능력을 테스트한다. 현재 LangChain 버전에서 A~D의 import 경로는 모두 실제로 존재하며, E의 변수명 오타도 런타임 에러를 유발하지 않는다.

> **참고:** 만약 출제 의도가 "과거 버전 기준"이라면, `create_agent`는 비교적 최근에 `langchain.agents`에 추가되었으며, 이전에는 `langgraph.prebuilt.create_react_agent`만 존재했다. 그 기준이라면 **(C)** 가 오류에 해당할 수 있다.

### Q10. Tool docstring 품질 — 정답: **(B)**

> Tool B — LLM은 docstring을 tool 선택의 근거로 사용하므로, 구체적일수록 정확도가 높다.

**해설:** LLM은 tool의 `description`(= docstring)과 파라미터 스키마를 보고 어떤 tool을 호출할지 결정한다.

- **Tool A** (`"""검색"""`) — 너무 추상적. "무엇을" 검색하는지, 어떤 인자가 필요한지 LLM이 판단하기 어려움
- **Tool B** — 함수명(`search_hospital`), 상세 docstring(지역+진료과목), Args/Returns 까지 명시 → LLM이 정확한 상황에 정확한 인자로 호출 가능

"토큰 절약"은 tool description에서 의미 있는 수준이 아니다. 정확도가 훨씬 중요하다.

### Q11. 코드 교정 — 3가지 버그

**Bug 1:** `self.checkpointer = MemorySaver()` — 인스턴스마다 새로운 `MemorySaver`가 생성됨

→ **수정:** 모듈 레벨 싱글턴으로 분리하거나 클래스 변수로 공유
```python
# 모듈 레벨 싱글턴
_checkpointer = MemorySaver()

class MedicalService:
    def __init__(self):
        self.checkpointer = _checkpointer
```

**Bug 2:** `@tool`을 클래스 메서드(`self` 포함)에 적용 — `self`가 tool 스키마에 노출됨

→ **수정:** 독립 함수로 분리
```python
@tool
def search_symptom(symptom: str) -> str:
    """증상으로 질병 정보를 검색합니다."""
    return f"{symptom} 관련 질병 정보"

class MedicalService:
    def __init__(self):
        self.tools = [search_symptom]
```

**Bug 3:** `response_fromat=ToolStrategy(ChatResponse)` — 키워드 인자명 오타 (`fromat` → `format`)

→ **수정:**
```python
response_format=ToolStrategy(ChatResponse)
```

### Q13. 설계 오류 설명

**문제점:** `process_query()`가 호출될 때마다 `InMemorySaver()`를 새로 생성하고 `create_agent()`도 새로 호출한다. 이로 인해:
1. 매 요청마다 **빈 메모리**로 시작 → 이전 대화를 전혀 기억하지 못함
2. `thread_id`를 전달해도, 해당 thread의 이력이 저장된 checkpointer 자체가 매번 사라지므로 의미 없음
3. 에이전트도 매번 재생성되므로 불필요한 초기화 비용 발생

**수정 코드:**

```python
class AgentService:
    def __init__(self):
        # checkpointer와 agent를 한 번만 생성 (싱글턴)
        self._checkpointer = InMemorySaver()
        self._agent = create_agent(
            model=self.model,
            tools=self.tools,
            checkpointer=self._checkpointer,
        )

    def process_query(self, query: str, thread_id: str) -> str:
        result = self._agent.invoke(
            {"messages": [{"role": "user", "content": query}]},
            {"configurable": {"thread_id": thread_id}},
        )
        return result["messages"][-1].content
```

**핵심:** `InMemorySaver`와 agent를 `__init__`에서 한 번만 생성하고, `process_query`에서는 재사용한다. 서로 다른 `thread_id`는 동일한 checkpointer 내에서 격리된 대화로 관리된다.

### Q15. 자유 서술 — `ToolStrategy` vs `@tool` 직접 정의

`response_format=ToolStrategy(ChatResponse)`는 에이전트의 **최종 응답 형식**을 강제하는 공식 API이다. 에이전트가 모든 tool 호출을 마치고 사용자에게 응답할 때, 반드시 `ChatResponse` 스키마에 맞춰 구조화된 응답을 생성하도록 보장한다. 반면, `ChatResponse`를 `@tool`로 정의하여 tools 목록에 추가하는 방식은 LLM이 이를 "선택 가능한 도구 중 하나"로 인식하므로, 반드시 호출한다는 보장이 없고 다른 tool과 혼동될 수 있다. 공식 API 관점에서 `ToolStrategy`가 권장되며, 그 이유는 응답 형식의 **일관성과 강제성**을 프레임워크 레벨에서 보장하기 때문이다. 우리 프로젝트에서도 `real_estate_agent.py`에서 `response_format=ToolStrategy(ChatResponse)`를 사용하고 있으며, `agent_service.py`가 tool_calls에서 `"ChatResponse"`라는 이름을 파싱하는 구조와 정확히 맞물린다.
