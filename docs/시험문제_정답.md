### Q1. `@tool` 데코레이터 적용 위치 — 정답: **(B)**

```python
@tool
def search_hospital(region: str) -> str:
    """병원을 검색합니다."""
    return f"{region} 병원 목록"
```

**해설:**

`@tool`은 쉽게 말하면 "이 함수를 LLM이 호출할 수 있게 등록해줘"라는 의미입니다.
데코레이터가 하는 일은 크게 3가지인데요:

1. 함수 **이름** → tool 이름으로 사용 (예: `search_hospital`)
2. 함수 **docstring** → LLM에게 보여줄 설명으로 사용 (예: `"병원을 검색합니다."`)
3. 함수 **파라미터 타입** → JSON Schema로 변환해서 LLM에게 전달 (예: `region: str` → `{"region": {"type": "string"}}`)

LLM은 이 정보를 받아서 "아, `search_hospital`이라는 도구가 있고, `region`에 지역 이름을 넣으면 되는구나" 하고 이해하게 됩니다.

**왜 나머지가 틀렸는지 보면:**

**(A) 클래스 메서드에 `@tool`을 쓴 경우:**
```python
class MedicalAgent:
    @tool
    def search_hospital(self, region: str) -> str:  # ← self가 문제
```
Python 클래스 메서드의 첫 번째 파라미터는 `self`입니다. `@tool`은 이 `self`도 "LLM이 값을 넣어야 하는 인자"로 인식해버립니다. 그러면 LLM에게 `{"self": {"type": "???"}, "region": {"type": "string"}}` 같은 스키마가 전달되는데, LLM은 `self`에 뭘 넣어야 하는지 모릅니다. 그래서 이상한 값을 넣거나 에러가 발생합니다.

**(C) 나중에 `tool()` 팩토리 호출:**
```python
def search_hospital(region: str) -> str:
    ...
tool(search_hospital)  # 반환값을 안 저장했음!
```
`tool(search_hospital)`은 새로운 Tool 객체를 **반환**하는 건데, 이걸 변수에 안 담았으니 그냥 날아갑니다. `search_hospital = tool(search_hospital)` 이렇게 해야 하는데, 어차피 이 패턴 자체가 공식 권장이 아닙니다.

**(D) `__init__` 안에서 클로저로 정의:**
동작은 할 수 있는데, 인스턴스를 만들 때마다 tool이 새로 정의되고 디버깅도 까다로워집니다. 공식 문서에서 권장하지 않는 패턴입니다.

**우리 프로젝트에서도** `app/agents/tools.py`를 보면 `@tool`이 전부 독립 함수에 붙어있습니다.

### Q2. Tool docstring의 역할 — 정답: **(B)**

> LLM이 어떤 Tool을 사용할지 결정하는 근거(메타데이터)로 사용된다.

**해설:**

보통 Python에서 docstring은 "개발자를 위한 주석" 정도이지만, LangChain에서 `@tool`과 함께 쓰이는 docstring은 완전히 역할이 다릅니다 — **LLM이 읽는 설명서**가 됩니다.

이 과정을 단계별로 보면:

**1단계 — Tool 등록 시점:**
```python
@tool
def search_apartment_trades(region: str, year_month: str) -> str:
    """아파트 매매 실거래가를 조회합니다.
    특정 지역의 아파트 매매 거래 내역을 조회할 때 사용합니다."""
```
`@tool` 데코레이터가 이 docstring을 뽑아서 Tool 객체의 `description` 필드에 저장합니다.

**2단계 — 에이전트에게 질문할 때:**
"강남구 아파트 가격 알려줘"라고 물으면, LangChain이 LLM에게 이런 정보를 보내줍니다:
```
사용 가능한 도구:
1. search_apartment_trades: "아파트 매매 실거래가를 조회합니다..."
2. search_apartment_rentals: "아파트 전월세 실거래가를 조회합니다..."
```

**3단계 — LLM이 판단:**
LLM이 사용자 질문("가격 알려줘")과 각 tool 설명을 비교해서, "이건 매매 가격 이야기니까 `search_apartment_trades`를 써야겠다" 하고 결정합니다.

**그러면 docstring이 엉성하면 어떻게 될까요?**
- `"""검색"""` 같은 모호한 설명이면 → LLM이 이걸 언제 써야 하는지 판단을 못 합니다
- docstring 자체가 없으면 → LangChain이 경고를 발생시킵니다

**나머지 보기가 틀린 이유:**
- **(A)** "그냥 문서화용 주석이다" → 일반 Python에서는 맞지만, LangChain `@tool`에서는 LLM이 실제로 읽고 판단하는 데이터입니다
- **(C)** "타입 검사에 쓰인다" → 타입 검사는 파라미터 힌트(`region: str`)가 담당하며, docstring과는 무관합니다
- **(D)** "UI 도움말" → LangChain은 UI 프레임워크가 아니라서, docstring이 사용자에게 직접 보이지 않습니다

### Q3. Checkpointer 싱글턴 관리 — 정답: **(B)**

> 코드 B — `MemorySaver()`를 매 호출마다 생성하면 `thread_id` 기반 대화 연속성이 파괴된다.

**해설:**

이걸 이해하려면 먼저 **MemorySaver가 무엇인지** 알아야 합니다.

`MemorySaver`는 **대화 내용을 저장하는 저장소**입니다. 카카오톡에 비유하면:
- `MemorySaver` = 카카오톡 앱 자체 (모든 채팅방 메시지 보관)
- `thread_id` = 각 채팅방 번호 (누구와의 대화인지 구분)

**코드 A (정상):**
```python
_checkpointer = MemorySaver()  # 모듈 레벨: 앱 시작할 때 딱 한 번 생성

def create_agent_instance(model, tools):
    return create_agent(model, tools, checkpointer=_checkpointer)
```
카카오톡 앱이 하나 있고, 모든 채팅방이 이 앱 안에 있는 구조입니다. `thread_id="user123"`으로 대화하면 이력이 `_checkpointer` 안에 쌓이고, 다음에 같은 `thread_id`로 오면 이전 대화를 기억합니다.

**코드 B (버그):**
```python
def create_agent_instance(model, tools):
    checkpointer = MemorySaver()  # 함수 호출마다 새로 생성!
    return create_agent(model, tools, checkpointer=checkpointer)
```
이건 **매번 카카오톡을 삭제하고 새로 설치하는 것**과 같습니다. 새 `MemorySaver()`는 텅 빈 저장소라서, `thread_id`가 같아도 이전 대화가 없습니다. "아까 강남구 물어봤잖아" 해도 "무슨 말씀인지 모르겠습니다"라고 나옵니다.

**(A) "여러 요청이 메모리를 공유하면 버그"라는 설명이 틀린 이유:**
"공유"라는 표현이 부정적으로 들릴 수 있는데, 사실 공유가 **필수**입니다. `thread_id`가 격리 역할을 하기 때문입니다:
- `thread_id="user_A"` → A의 대화만 보관
- `thread_id="user_B"` → B의 대화만 보관
같은 `MemorySaver` 안에 있어도 서로 다른 `thread_id`의 대화는 절대 섞이지 않습니다.

**실제로 이 버그를 경험했습니다:**
우리 프로젝트 `chat.py`에서 정확히 코드 B와 같은 문제가 있었습니다. `AgentService()`를 요청마다 새로 만들어서 "이전 대화 내용을 기억하지 못합니다" 버그가 발생했고, 모듈 레벨 싱글턴으로 수정하여 해결했습니다 (커밋 `ac3b0fc`).

### Q4. 올바른 import 경로 — 정답: **(B), (E)**

**해설:**

이걸 이해하려면 LangChain의 패키지 구조를 알아야 합니다:

```
langchain-core    ← 핵심 요소들 (messages, tools, prompts 등)
langchain         ← 통합 레이어 (agents, chains 등 + core에서 가져온 것 재수출)
langchain-openai  ← OpenAI 연동
langgraph         ← 그래프 기반 에이전트 런타임
```

**핵심 원칙: `langchain_core`에 있는 것은 `langchain_core`에서 직접 가져오는 것이 권장됩니다.**

| 보기 | 실제 위치 | 판정 | 설명 |
|------|-----------|------|------|
| **(A)** `from langchain.tools import tool` | `langchain_core.tools`에서 재수출된 것 | △ 되긴 하지만 비권장 | `langchain`이 `langchain_core` 것을 그대로 다시 내보낸 것이라, 원본에서 직접 가져오는 게 낫습니다 |
| **(B)** `from langchain_core.tools import tool` | 원본 위치 | **✓ 권장** | `tool` 데코레이터가 실제로 있는 곳입니다 |
| **(C)** `from langchain.agents.structured_output import ToolStrategy` | `langchain.agents` 하위 | △ 되긴 함 | 우리 프로젝트에서도 쓰고 있고 동작하지만, `langchain_core` 대안이 있다면 그쪽이 더 안정적입니다 |
| **(D)** `from langchain.agents import create_agent` | `langchain.agents` 하위 | △ 되긴 함 | 우리 프로젝트에서 쓰고 있지만, 버전에 따라 가용성이 다릅니다 |
| **(E)** `from langchain_core.messages import HumanMessage, SystemMessage` | 원본 위치 | **✓ 권장** | 메시지 클래스의 정식 경로입니다 |

**왜 `langchain_core`가 권장될까요?**

`langchain` 패키지는 버전 업데이트가 빠르고 내부 구조가 자주 바뀝니다. 반면 `langchain_core`는 **안정적인 핵심 API**를 제공하는 패키지라서 하위 호환성이 더 잘 보장됩니다.

비유하면 `langchain.tools.tool`은 "편의점에서 산 생수", `langchain_core.tools.tool`은 "정수기에서 직접 받은 물"입니다. 같은 물인데, 원본에서 직접 가져오는 게 더 확실합니다.

### Q5. `create_agent`의 `system_prompt` 파라미터 — 정답: **(B)**

> 문자열(`str`) 또는 `SystemMessage` 객체 모두 전달 가능하다.

**해설:**

**System Prompt란?**
에이전트에게 "너는 이런 역할이고, 이렇게 행동해"라고 알려주는 지시문입니다. ChatGPT에서 Custom Instructions를 설정하는 것과 같은 개념입니다.

예를 들어 우리 프로젝트의 system prompt:
```
"너는 부동산 실거래가 분석 전문 에이전트다. 사용자가 지역명과 기간을 말하면
적절한 Tool을 호출해서 데이터를 조회하고 분석 결과를 줘라..."
```

**전달 방법이 2가지:**

```python
# 방법 1: 문자열 — 간편하고 직관적
agent = create_agent(
    model=model, tools=tools,
    system_prompt="너는 부동산 분석 에이전트다."  # ← str
)

# 방법 2: SystemMessage 객체 — 더 명시적
from langchain_core.messages import SystemMessage
agent = create_agent(
    model=model, tools=tools,
    system_prompt=SystemMessage(content="너는 부동산 분석 에이전트다.")
)
```

문자열을 넘기면 내부에서 자동으로 `SystemMessage`로 변환해주기 때문에 결과는 동일합니다. 편하게 문자열을 쓰면 됩니다.

**나머지가 틀린 이유:**
- **(A)** `ChatPromptTemplate` 객체를 반드시 전달해야 한다 → 과거 방식이고, `create_agent`는 더 단순한 인터페이스를 제공합니다
- **(C)** 리스트만 된다 → "만"이 틀립니다. 리스트도 되지만 문자열도 됩니다
- **(D)** 파라미터가 없다 → 있습니다. 우리 프로젝트에서 쓰고 있습니다

**프로젝트 근거:** `real_estate_agent.py:62`에서 `system_prompt=system_prompt`로 문자열을 직접 전달하고 있습니다.

### Q6. `response_format`과 `ToolStrategy` — 정답: **(B)**

> 방법 B가 올바르다. `response_format=ToolStrategy(ChatResponse)`가 공식 API다.

**해설:**

이 문제의 핵심은 "에이전트가 최종 응답을 어떤 형태로 돌려주느냐"의 차이입니다.

**왜 구조화된 응답이 필요한가?**
우리 프론트엔드는 에이전트 응답을 파싱해서 코드 블록, 차트, 테이블로 렌더링합니다. 그러려면 에이전트가 그냥 텍스트가 아니라 `{"message_id": "...", "content": "...", "metadata": {...}}` 같은 정해진 구조로 응답해야 합니다.

**방법 A의 문제 — `ChatResponse`를 `@tool`로 등록:**
```python
agent = create_agent(model, tools=[search, ChatResponse])
```
이렇게 하면 `ChatResponse`가 `search`와 동등한 "도구 중 하나"로 등록됩니다. LLM 입장에서는:
- `search` — 검색하는 도구
- `ChatResponse` — 응답하는 도구

문제는 LLM이 때때로 `ChatResponse`를 안 쓰고 그냥 텍스트로 답해버리거나, `search` 대신 `ChatResponse`를 호출할 수 있다는 것입니다. **강제성이 없습니다.**

**방법 B (정답) — `response_format=ToolStrategy(ChatResponse)`:**
```python
agent = create_agent(
    model, tools=[search],
    response_format=ToolStrategy(ChatResponse)
)
```
에이전트에게 "최종 응답은 반드시 `ChatResponse` 형식이어야 해"라고 **프레임워크 레벨에서 강제**하는 것입니다. LLM이 무슨 작업을 하든 마지막에는 무조건 이 양식으로 대답해야 합니다.

**비유:**
- 방법 A = 뷔페에서 "디저트도 있으니 원하시면 드세요" (안 먹을 수도 있음)
- 방법 B = 코스 요리에서 "마지막에 디저트가 무조건 나옵니다" (거부 불가)

**프로젝트 근거:** `real_estate_agent.py:63`에서 정확히 방법 B를 사용합니다. 그리고 `agent_service.py`에서 tool_calls에서 `"ChatResponse"` 이름을 찾아 파싱하는 로직이 있는데, `ToolStrategy`가 이 호출을 보장하니까 파싱이 항상 성공합니다.

### Q7. 클래스 메서드와 `@tool` — 정답: **(B)**

> `self` 파라미터가 LangChain tool 스키마에 노출되어 LLM이 `self`를 인자로 채우려 시도한다.

**해설:**

Q1에서도 나온 내용인데, 좀 더 구체적으로 살펴보겠습니다.

**Python 클래스 메서드 기본:**
```python
class StockAgent:
    def get_stock_price(self, ticker: str) -> str:
        return f"{ticker}: 50000원"
```
`self`는 "이 메서드를 호출한 객체 자신"을 가리키는 파라미터입니다. `agent.get_stock_price("삼성전자")`를 호출하면 Python이 자동으로 `self=agent`를 넣어줍니다. 개발자가 직접 넘기지 않아도 됩니다.

**그런데 `@tool`을 붙이면?**
```python
class StockAgent:
    @tool
    def get_stock_price(self, ticker: str) -> str:
        """주식 현재가를 조회합니다."""
```
`@tool`은 함수의 **모든 파라미터**를 분석해서 JSON Schema를 만듭니다. `self`가 Python에서 특별한 의미를 가진다는 것을 **모릅니다.** 그래서 이런 스키마가 나옵니다:

```json
{
  "name": "get_stock_price",
  "parameters": {
    "self": {"type": "object"},   // ← LLM: "이게 뭐지...?"
    "ticker": {"type": "string"}
  }
}
```

LLM은 `self`에 뭘 넣어야 하는지 모르기 때문에:
- 이상한 값(`"StockAgent"`, `{}` 등)을 넣거나
- 호출 자체를 포기하거나
- 런타임 에러가 발생합니다

**해결법:**
```python
# 독립 함수로 분리하면 됩니다 (권장)
@tool
def get_stock_price(ticker: str) -> str:
    """주식 현재가를 조회합니다."""
    return f"{ticker}: 50000원"
```

우리 프로젝트 `tools.py`에서도 모든 `@tool` 함수가 독립 함수로 되어있습니다. 클래스 메서드에 `@tool`을 쓴 적이 없습니다.

### Q8. Checkpointer와 멀티턴 대화 — 정답: **(B)**

> `create_agent()`에 `checkpointer` 파라미터로 `MemorySaver()` 등 전달 + 각 invoke에 `{"configurable": {"thread_id": "..."}}` 전달

**해설:**

**멀티턴 대화란?**
여러 번 주고받는 대화를 말합니다. 예시:
```
사용자: 강남구 아파트 가격 알려줘        ← 1턴
에이전트: 강남구 최근 거래는...
사용자: 그중에서 10억 이상만 골라줘       ← 2턴 ("그중에서"가 1턴을 기억해야 함)
```

2턴에서 "그중에서"라고 했을 때, 1턴의 "강남구 아파트"를 기억해야 대화가 성립합니다.

**멀티턴에 필요한 2가지:**

**1. Checkpointer — "어디에" 대화를 저장할 것인지**
```python
checkpointer = InMemorySaver()  # 메모리(RAM)에 저장
agent = create_agent(model, tools, checkpointer=checkpointer)
```

**2. thread_id — "어떤 대화"인지 구분하는 키**
```python
result = agent.invoke(
    {"messages": [{"role": "user", "content": "강남구 아파트 가격"}]},
    {"configurable": {"thread_id": "user_123_session_1"}}
)
```
같은 `thread_id`로 다시 호출하면 이전 대화를 이어갑니다. 다른 `thread_id`면 새 대화입니다.

**비유:**
- `checkpointer` = 학교 사물함 전체 (모든 학생 물건 보관)
- `thread_id` = 각 학생의 사물함 번호 (자기 것만 꺼냄)

사물함(`checkpointer`)이 없으면 물건을 보관할 곳이 없고, 번호(`thread_id`)가 없으면 어떤 사물함을 열어야 할지 모릅니다. **둘 다 있어야** 합니다.

**나머지가 틀린 이유:**
- **(A)** `memory=True` 플래그 → 이런 파라미터 자체가 없습니다
- **(C)** 이전 메시지를 직접 리스트에 추가 → 기술적으로 되긴 하지만, 대화가 길어지면 토큰이 폭발하고 매 요청마다 전체 이력을 보내야 합니다. checkpointer가 이걸 자동 관리해줍니다
- **(D)** 멀티턴 지원 안 한다 → 합니다. 우리 프로젝트가 증거입니다

**프로젝트 근거:**
- `real_estate_agent.py:64` → `checkpointer=checkpointer`
- `agent_service.py` → `{"configurable": {"thread_id": thread_id}}`

### Q9. AI Hallucination 식별 — 정답: 해당 없음 (현재 버전에서 모두 정상)

**해설:**

**AI Hallucination(환각)이란?**
AI가 존재하지 않는 것을 마치 사실인 것처럼 그럴듯하게 말하는 현상입니다. 코딩에서 흔한 예:
- 없는 라이브러리 이름을 알려줌 (`pip install magic-solver`)
- 없는 import 경로를 알려줌 (`from langchain.magic import SuperAgent`)
- 없는 함수나 파라미터를 사용함

이 문제는 "아래 코드 중 AI가 날조한 것이 무엇인가?"를 묻고 있습니다.

**각 보기를 하나씩 보면:**

| 보기 | 코드 | 동작 여부 | 설명 |
|------|------|-----------|------|
| **(A)** | `from langchain.agents.structured_output import ToolStrategy` | ✅ 동작 | 우리 프로젝트 `real_estate_agent.py:13`에서 실제로 쓰고 있습니다 |
| **(B)** | `from langchain_core.tools import tool` | ✅ 동작 | 공식 canonical 경로입니다. 가장 정석적인 import입니다 |
| **(C)** | `from langchain.agents import create_agent` | ✅ 동작 | 우리 프로젝트 `real_estate_agent.py:12`에서 실제로 쓰고 있습니다 |
| **(D)** | `from langchain_core.messages import HumanMessage` | ✅ 동작 | 공식 canonical 경로입니다 |
| **(E)** | `response_fromat = ToolStrategy(ChatResponse)` | ✅ 문법상 동작 | `response_fromat`은 변수명에 오타(`fromat` → `format`)가 있는데, Python에서 변수명은 자유이기 때문에 에러가 나지 않습니다. `response_format=response_fromat`은 **키워드 인자** `response_format`에 **변수** `response_fromat`의 값을 넣는 것이라 정상 동작합니다 |

**결론:** 현재 우리 프로젝트에서 쓰는 LangChain 버전 기준으로 A~E 전부 에러가 나지 않습니다.

**다만 출제 의도를 추측해보면:**
과거 LangChain 버전 기준이라면:
- `create_agent`는 비교적 최근에 `langchain.agents`에 추가된 것이고, 예전에는 `from langgraph.prebuilt import create_react_agent`밖에 없었습니다. 그 기준이면 **(C)** 가 오류에 해당합니다
- `ToolStrategy`도 비교적 최근 추가라서 **(A)** 도 해당될 수 있습니다

**이 문제가 가르치는 교훈:** AI가 알려주는 import 경로를 맹신하지 말고, 반드시 공식 문서나 실제 실행으로 검증해야 합니다.

### Q10. Tool docstring 품질 — 정답: **(B)**

> Tool B — LLM은 docstring을 tool 선택 근거로 사용하므로, 구체적일수록 정확도가 높다.

**해설:**

Q2에서 설명한 것처럼 LLM은 tool의 docstring을 보고 "이걸 지금 써야 하나?" 판단합니다. 두 tool을 비교해보겠습니다:

**Tool A — 대충 쓴 docstring:**
```python
@tool
def search(q: str) -> str:
    """검색"""
```
LLM에게 전달되는 정보:
```
도구: search / 설명: "검색" / 파라미터: q (문자열)
```
LLM의 생각: "검색... 무엇을 검색하라는 건지? 웹? DB? 파일? `q`에 뭘 넣어야 하지?"
→ 잘못된 상황에 호출하거나, 필요한 상황에 호출 안 할 확률이 높습니다

**Tool B — 상세하게 쓴 docstring:**
```python
@tool
def search_hospital(region: str, specialty: str = "내과") -> str:
    """지역과 진료과목으로 병원을 검색합니다.

    Args:
        region: 검색할 지역명 (예: '서울', '강남구')
        specialty: 진료과목 (예: '내과', '정형외과')
    """
```
LLM의 생각: "병원 검색 도구구나. region에 '강남구', specialty에 '정형외과'를 넣으면 되겠다."
→ 정확한 상황에 정확한 인자로 호출합니다

**(A) "토큰 절약이 더 효율적"이라는 말이 틀린 이유:**
tool description은 보통 수십~수백 토큰 수준입니다. LLM 컨텍스트 윈도우(수만~수십만 토큰)에 비하면 극히 미미한 양인데, 토큰 몇 개 아끼려다 정확도를 떨어뜨리는 것은 좋지 않습니다.

우리 프로젝트에서도 `search_apartment_trades`와 `search_apartment_rentals`에 상세한 docstring을 써놨기 때문에 LLM이 "매매"와 "전월세"를 정확히 구분해서 호출합니다.

### Q11. 코드 교정 — 3가지 버그

**원본 코드:**
```python
from langchain.agents import create_agent
from langchain_core.tools import tool

class MedicalService:
    def __init__(self):
        self.checkpointer = MemorySaver()  # bug 1

    @tool  # bug 2
    def search_symptom(self, symptom: str) -> str:
        """증상으로 질병 정보를 검색합니다."""
        return f"{symptom} 관련 질병 정보"

    def create(self, model):
        return create_agent(
            model,
            tools=[self.search_symptom],
            response_fromat=ToolStrategy(ChatResponse)  # bug 3
        )
```

---

**Bug 1: `self.checkpointer = MemorySaver()` — 인스턴스마다 새 MemorySaver 생성**

**문제:**
`MedicalService()`를 여러 곳에서 만들면 (`s1 = MedicalService()`, `s2 = MedicalService()`) 각각 별도의 `MemorySaver()`가 생깁니다. Q3에서 설명한 것처럼 대화 이력이 공유되지 않아 멀티턴 대화가 깨집니다.

**수정:**
```python
# 모듈 레벨에서 딱 한 번만 생성 (서버가 살아있는 동안 유지)
_checkpointer = MemorySaver()

class MedicalService:
    def __init__(self):
        self.checkpointer = _checkpointer  # 같은 인스턴스를 참조
```

---

**Bug 2: 클래스 메서드에 `@tool` — `self`가 노출됨**

**문제:**
```python
@tool
def search_symptom(self, symptom: str) -> str:
```

**수정:**
```python
# 클래스 밖으로 빼서 독립 함수로 만듭니다
@tool
def search_symptom(symptom: str) -> str:
    """증상으로 질병 정보를 검색합니다."""
    return f"{symptom} 관련 질병 정보"

class MedicalService:
    def __init__(self):
        self.tools = [search_symptom]
```

response_format=ToolStrategy(ChatResponse)  # "format" 올바른 철자

### Q12. 빈칸 채우기

> `create_agent()`의 `system_prompt` 파라미터는 **`str`** 또는 **`SystemMessage`** 타입을 받을 수 있다.

> 멀티턴 대화를 유지하려면 `create_agent()`에 `checkpointer=`**`InMemorySaver`**`()` 와 같이 서버 전체에서 단 한 번 생성된 인스턴스를 전달해야 한다.

**해설:**

**첫 번째 빈칸 — `str` 또는 `SystemMessage`:**
Q5에서 설명한 내용입니다. `create_agent()`는 시스템 프롬프트를 두 형태로 받습니다:

- `SystemMessage`: `SystemMessage(content="...")` — 더 명시적입니다

문자열을 넘기면 내부에서 자동으로 `SystemMessage`로 변환해주기 때문에 결과는 동일합니다.

`InMemorySaver`는 `langgraph.checkpoint.memory`에서 제공하는 체크포인터입니다. 이름 그대로 RAM에 대화 상태를 저장합니다. 서버가 꺼지면 날아갑니다.

핵심은 **"서버 전체에서 단 한 번"** 이라는 조건입니다. Q3에서 봤듯이 매번 새로 만들면 이전 대화가 사라집니다.

**프로젝트 근거:**
- `prompts.py`에서 `system_prompt`를 `str`로 정의 → `real_estate_agent.py:62`에서 문자열로 전달
- `agent_service.py:26`에서 `InMemorySaver()`를 한 번만 생성

### Q13. 설계 오류 설명

**원본 코드:**
```python
class AgentService:
    def process_query(self, query: str, thread_id: str) -> str:
        checkpointer = InMemorySaver()  # ← 여기가 문제
        agent = create_agent(
            model=self.model, tools=self.tools, checkpointer=checkpointer
        )
        result = agent.invoke(
            {"messages": [{"role": "user", "content": query}]},
            {"configurable": {"thread_id": thread_id}}
        )
        return result["messages"][-1].content
```

**문제점:**

일상에 비유하면 **"고객이 전화할 때마다 새 전화기를 사서 통화하고, 끝나면 전화기를 버리는 콜센터"** 입니다.

`process_query()`가 호출될 때마다:
1. `InMemorySaver()` — 새 빈 저장소 생성 (이전 대화 없음)
2. `create_agent()` — 새 에이전트 생성 (모델, 도구 재초기화)
3. `agent.invoke()` — 빈 상태에서 대화 시작

그 결과:
- **대화 연속성 파괴:** `thread_id`를 같은 값으로 줘도, 새 `InMemorySaver`에는 이전 대화가 없으니 의미가 없습니다. "아까 강남구 물어봤잖아" → "무슨 말씀인지 모르겠습니다"
- **성능 낭비:** 에이전트 생성(모델 초기화, 도구 바인딩)은 비용이 있는 작업인데 매번 반복됩니다
- **thread_id가 무의미:** thread_id의 존재 이유가 "같은 ID면 이전 대화 이어가기"인데, 저장소 자체가 매번 리셋되니 이어갈 것이 없습니다

**수정 코드:**

```python
class AgentService:
    def __init__(self):
        # ✅ __init__에서 한 번만 생성
        self._checkpointer = InMemorySaver()
        self._agent = create_agent(
            model=self.model,
            tools=self.tools,
            checkpointer=self._checkpointer,
        )

    def process_query(self, query: str, thread_id: str) -> str:
        # ✅ 기존 agent를 재사용 — 새로 만들지 않음
        result = self._agent.invoke(
            {"messages": [{"role": "user", "content": query}]},
            {"configurable": {"thread_id": thread_id}},
        )
        return result["messages"][-1].content
```

**수정의 핵심:**
- `InMemorySaver()`를 `__init__`에서 한 번만 생성 → 모든 `process_query()` 호출이 같은 저장소를 공유합니다
- `create_agent()`도 `__init__`에서 한 번만 호출 → 에이전트를 재사용합니다
- 서로 다른 `thread_id`는 같은 checkpointer 안에서 격리된 대화로 관리됩니다

**중요한 추가사항:**
`AgentService` 자체도 **한 번만 만들어야** 합니다. 요청마다 `AgentService()`를 새로 만들면 `__init__`이 매번 호출되어 같은 문제가 반복됩니다:
```python
# chat.py (라우터)
_agent_service = AgentService()  # 모듈 레벨: 서버 시작 시 한 번만

@chat_router.post("/chat")
async def chat(request: ChatRequest):
    return await _agent_service.process_query(request.message, request.thread_id)
```

**우리 프로젝트에서 실제로 이 버그를 경험하고 수정했습니다** (커밋 `ac3b0fc`).

### Q15. 자유 서술 — `ToolStrategy` vs `@tool` 직접 정의

**답변:**

LangChain에서 에이전트의 구조화된 응답을 만드는 방법이 2가지 있는데, 이 둘은 **근본적으로 다른 목적**을 가집니다.

**`response_format=ToolStrategy(ChatResponse)` — 응답 형식 강제:**

에이전트에게 "최종 응답은 반드시 `ChatResponse` 형식으로 줘"라고 프레임워크 레벨에서 강제하는 것입니다. 에이전트가 검색이든 계산이든 무슨 작업을 하든, 마지막에 사용자에게 답할 때는 무조건 `ChatResponse` 스키마(`message_id`, `content`, `metadata`)에 맞춰야 합니다. 거부할 수 없습니다.

**`ChatResponse`를 `@tool`로 정의 — 선택적 도구 등록:**

`ChatResponse`를 `search`와 같은 "도구 중 하나"로 등록하는 것입니다. LLM 입장에서는 쓸 수도 있고 안 쓸 수도 있습니다. 그냥 텍스트로 답해버릴 수도 있고, 다른 도구와 혼동할 수도 있습니다. 강제성이 없습니다.

**어느 쪽이 권장되는가?**

당연히 `ToolStrategy`입니다. 이유는:

1. **일관성:** 모든 응답이 반드시 동일한 구조를 따르므로 프론트엔드 파싱이 안정적입니다
2. **강제성:** LLM의 "기분"에 의존하지 않고 프레임워크가 형식을 보장합니다
3. **관심사 분리:** "도구"(검색, 계산 등)와 "응답 형식"(출력 구조)이 깔끔하게 분리됩니다. `@tool`은 도구용이고 `response_format`은 출력 형식용입니다

우리 프로젝트도 `real_estate_agent.py`에서 `response_format=ToolStrategy(ChatResponse)`를 사용하고 있고, `agent_service.py`가 tool_calls에서 `"ChatResponse"` 이름을 찾아서 파싱하는 구조입니다. `ToolStrategy`가 이 호출을 보장하기 때문에 프론트엔드는 항상 예측 가능한 구조의 데이터를 받을 수 있습니다.
